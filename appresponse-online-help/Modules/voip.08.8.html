Configuring the Module for VoIP Performance
After successfully installing the package, complete the following steps for general configuration and customization of the Module for VoIP Performance Dashboard:
1)
Access the Module for VoIP Performance Web UI and log in with an Administrator account.
2)
Obtain and enter a VoIP License Key.
3)
Verify monitor interface configuration.
4)
Verify VoIP signaling port configuration.
5)
Add Business Groups (if desired).
The following sections describe these steps in detail.
Note: Steps can be completed at your discretion; there is a default series of metrics at initial start-up.
Troubleshooting information is located in Administration Tools.
Protocol Settings Page
The Protocol Settings page contains two sets of controls. To go to the Protocol Settings page, choose Administration Tools > Protocol Settings in the Module for VoIP Performance menu.
Figure 4 Administration Tools, Protocol Settings
 
Protocol Handling Settings
These settings let you to change which ports VoIP signaling traffic is parsed for. You can use the defaults, but in some cases the ports might need to be adjusted depending on your environment. For example: some Vonage phone adapters use UDP port 10000 for signaling. Thus, when monitoring a network using these adapters, you would include 10000 in the SIP UDP ports entry.
Note: If SIP signaling is found on alternate ports, you must enter the ports in both the SIP UDP and SIP TCP fields.
For more information, see How to Enable SIP/TLS Decryption.
The Call Management Options section has the following options:
Enable Avaya Heuristics
This is an experimental feature and is provided to better support Avaya deployments that use proprietary signalling. Note the following:
–
This feature is not necessary for all workflows that involve analyzing VoIP quality through the business group metrics. This includes the VoIP business group metrics for monitoring (Dashboards) and Alerting. These workflows always work, regardless of the state of this feature.
–
This feature is only valuable for detailed call troubleshooting workflows, where calls need to be examined/searched by phone numbers.
–
This feature is not compatible with other telephony systems that use the H323 protocol. In a mixed vendor environment where this is enabled, the H323 call records for the non-Avaya call records will be improperly handled and will appear to be incomplete.
–
This feature is not helpful in an Avaya deployment configured to use SIP signaling.
Ignore Discards Due to Jitter
By default, VoIP Monitoring Module considers discards due to jitter when calculating MOS metrics. When this option is selected, VoIP Monitoring Module ignores discards due to jitter (but still considers jitter itself) when calculating MOS metrics. Enabling this option usually results in higher MOS metrics.
Note—Riverbed does not recommend enabling this option. Discards are an important factor when calculating MOS. You should enable this option only in environments where discards are not helpful to the analysis being performed.
Debug settings
These settings should not be changed except as instructed by Riverbed Technical Support.
 
Data Purge Page
The Administration Tools > Data Purge page enables you to modify how AppResponse retains and purges data when the appliance starts to run out of disk space.
The options on this page offer some control over how AppResponse allocates the disk space that is available for VoIP data. Call Summary data is generally the most helpful to retain because it includes statistic and channel data for individual calls. Time Series and Signalling data is generally useful only for troubleshooting one or more recent calls. Thus, you might want to retain more Call Summary data by specifying shorter retention times for Time Series and Signalling data.
Note the following:
The effective time window for VoIP data retention is also influenced by core AppResponse operations and processes. Do not change the default settings in this page unless you clearly know the specific settings you want.
The retention windows specified in this page are not absolute. They are enforced only when the appliance starts to run out of VoIP disk space and needs to purge data. If your call volume is low, the effective retention time will be longer than these thresholds; if your call volume is very high, the effective retention time will be shorter.
Module for VoIP Performance checks the amount of available disk space once per day (“Run daily at” fields, which use a 24-hour clock). If disk availability is low, Module for VoIP Performance purges VoIP data that is older than the thresholds specified in the “Retain call summary data,” “Retain call quality time series data,” and “Retain call signalling data” fields.
