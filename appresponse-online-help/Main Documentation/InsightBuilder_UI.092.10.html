Best Practices
Do not overlook these best practices! They can save you a lot of time.
Planning
Sketch first. You'll find that you make fewer mistakes and overlook fewer needed Interactions if you make a rough sketch of how you want the Insight to look before you start.
Saving and Versioning
Save early and often. There's no Undo function.
To prevent losing a lot of time if you take a wrong turn while constructing an Insight, consider versioning your saves every so often, so you have a point to return to and start again. This is most easily done from the Preview Insight Content window (invoked by clicking Test/Layout), which always asks you whether you want to overwrite the existing version of the Insight you’re working on.
Fill out the Insight Properties Window
Do this at the beginning of creating your Insight.
Follow a consistent work pattern
It's best to follow a consistent work pattern: either completing each row or each column before adding more rows or columns. Adding lots of rows and columns prevents saving until you're completely finished adding Components, which increases the chance of losing your work if something goes wrong.
Always Rename Components as you Create Them and Use a Consistent Convention
Always rename Components as you create them rather than accepting the numeric-increment default, and follow a consistent naming convention. It doesn't matter what convention you follow, as long as you're consistent. We suggest using colons to separate.
Note: Note: When renaming Hidden Visual Components, we suggest you use the word “hidden” in the name.
For instance, if you use a Favorites Table Component to display applications, rather than leaving it named Table:favorite1, name it something like Table:Applications. If a Multi-Group Chart Component is primarily intended to display Throughput, then name it Chart:multigrp:Throughput. It is useful to keep the Component type, in the Component name, i.e., a table is Table:Applications.
This is especially important given that Interaction’s names are automatically generated from the names given to Components.
Build Insights on an Appliance and Test on a Director
Always build an Insight on an Appliance unless it uses Director groups. An Insight built on a Director does not run on an Appliance.
Always Add a Box Control
The Box Toolbar Control is necessary if you want your Insight to be able to run on a Director; otherwise the Insight can't select which Appliance to pull data from when necessary.
Consider Adding an Initializer
Always put an Initializer Analysis Component in an Insight so that it can be linked to from another Insight, if you decide to do so in the future.
