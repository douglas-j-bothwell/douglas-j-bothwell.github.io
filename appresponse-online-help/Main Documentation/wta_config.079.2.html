Tracking Page Views and Page Families: Configuration Options
AppResponse includes a set of options that enable you to customize how Web Transaction Analysis monitors web traffic and collects relevant data.
The following options are available in the Administration > System > Pages web page:
Truncate Page Family URLs
Group Page Views Based on URL Parameters
Page Separation
The following options are available in the CLI. To configure any of these options, log in to the appliance as Administrator using an SSH-enabled utility such as putty.
Group SOAP Transactions into Separate Page Views
Remove Session IDs Embedded in URLs
Specify the HTTP Header Field that Contains the Originating IP
Do Not Reject Pages Based on Content Type
Note the following:
You must have Administrator privileges on the appliance to run any of these commands.
These commands affect only web traffic observed after the command is applied, and do not change any web traffic that has already been observed and stored on the appliance.
Each command includes a delete option to undo the command and revert to the default Web Transaction Analysis behavior.
After you change the Web Transaction Analysis configuration, the appliance might need to observe web traffic for several minutes before any changes become visible in the Desktop Console.
Truncate Page Family URLs
The Truncation settings define how the appliance truncates page family names that appear in the Individual Page Views insight and other charts, tables, and UI elements.
These options are available under Administration > System > Pages > Advanced Page Analysis Configuration Parameters > Truncation.
Figure 1
Truncation Options for Page Families
 
You can configure the appliance to truncate all characters at or after the first instance of:
The nth slash (after http://). If the appliance is configured to truncate at the fourth slash, the result is
http://www.example.com/a/b/c/d/e ==>
http://www.example.com/a/b/c/...
A custom character (if enabled). If the appliance is configured to truncate at the semicolon (;), the result is
http://www.example.com/a;bcd ==>
http://www.example.com/a...
Note: By default, the appliance truncates each URL at the question mark (?) that separates the resource path from the parameters, or at the fourth slash in the resource path (after http://), whichever comes first.
Group Page Views Based on URL Parameters
This option is useful when you want to create separate page families based on web application parameters passed as key=value strings in URLs. You can specify one or more parameter keys to ensure that page views that are “identical,” but have different values for the specified keys, are grouped into separate page families.
This option enables you to override the default behavior, which is to truncate all URL parameters after the delineating question mark (?) and to group all pages with the same truncated URL into the same family.
This option is available under Administration > System > Pages > Advanced Page Analysis Configuration Parameters > URL Parameters.
Figure 2
URL Parameter Options for Page Families
 
In the previous figure, the appliance is configured to create page families based on location and department parameters (location and department keys). The appliance creates separate families for the “same” page based on the parameter values observed in different page views:
   http://www.example.com/params...!loc=boston!dept=it
   http://www.example.com/params...!loc=nyc!dept=accnt
   http://www.example.com/params...!loc=london!dept=sales
Note the following:
You should order the table from highest-priority parameter (first entry in table) down. The appliance creates page families based on the order of key-value pairs in the table, regardless of where the parameters are located in the URLs.
You should include a “short” label to display for each parameter key.
Page Separation
This option is useful in situations when the appliance cannot separate multiple page views and incorrectly combines them into one view. You can specify a URL string so that any page view whose URL matches this string will always be considered a separate page.
This option is available under Administration > System > Pages > Advanced Page Analysis Configuration Parameters > Page Separation.

Group SOAP Transactions into Separate Page Views
This option is useful for analyzing SOAP transactions. When this option is enabled, the appliance does the following:
Translates every discrete SOAP transaction into a separate Page View (without SOAP processing enabled, a page view might include two or more SOAP transactions)
Includes the SOAP method in the URLs shown for page families and individual page views in the Page Analysis insights (for example, http://webapp.com/TradeProcess/Process.asmx [method:ProcessTrade])
Note the following:
This option requires additional computing resources and might slow down performance on the appliance.
Normally, you should enable this option only if your appliance is monitoring back-end SOAP traffic and not front-end browser traffic.
You cannot include SOAP methods in URLs when you define web applications (Desktop Console > Tools > Applications > Defined Application Manager > URL tab).
To enable this option, enter the following command:
WTA-StitcherConfig add soap-processing
To disable this option and revert to the default (ignore SOAP when constructing page views/families), enter the following command:
WTA-StitcherConfig delete soap-processing
Remove Session IDs Embedded in URLs
URLs that include session IDs can dramatically increase the number of page families. With this option enabled, the appliance removes from the URL many session ID strings used by common web application frameworks. Enabling this option can result in a smaller—and more easily navigable—set of page families.
Note the following:
This option requires additional computing resources and can slow down performance on the appliance.
This option removes only strings found in commonly-used web application frameworks such as ASP.net and Jsession.
To enable this option, enter the following:
WTA-StitcherConfig add common-session-id-removal
To disable this option and revert to the default (URLs include full session IDs), enter the following command:
WTA-StitcherConfig delete common-session-id-removal
Specify the HTTP Header Field that Contains the Originating IP
If a proxy or load balancer changes or hides IPs, the appliance reads the X-Forwarded-for (XFF) field in HTTP headers to determine the Originating IP. If the web traffic in your network embeds the Originating IP in a different header field, you can use this option to specify this field.
Note: If you enable this option, the appliance will look in the specified field only for the Originating IP; it will not look in the X-Forwarded-for (XFF) in addition to the specified field.
To enable this option, enter the following:
WTA-StitcherConfig add globals.stitching.originating-ip-http-field field
For example:
WTA-StitcherConfig add globals.stitching.originating-ip-http-field x-oip
To disable this option and revert to the default (look in X-Forwarded-for field for Originating IP), enter the following command:
WTA-StitcherConfig delete globals.stitching.originating-ip-http-field
Convert URLs to All-Lowercase
This option is useful when the appliance monitors page view URLs that are identical except for differences in case (uppercase/lowercase). By default, if two URLs have different cases but are otherwise identical, they will be grouped into separate page families.
To enable this option, log in to the CLI as Administrator and enter the following command:
WTA-StitcherConfig -s4 add page-family-urls.force-lowercase true
To disable this option and revert to the default behavior, enter the following command:
WTA-StitcherConfig delete page-family-urls.force-lowercase
 
Do Not Reject Pages Based on Content Type
This option is useful when you want to monitor page views that have unusual file types. By default, the appliance recognizes only page views that correspond to commonly recognized web-page file types such as *.htm, *.html, *.asp, and so on.
To enable this option, log in to the CLI as Administrator and enter the following command:
WTA-StitcherConfig -s4 add do-not-reject-pages-based-on-content-type
To disable this option and revert to the default behavior, enter the following command:
WTA-StitcherConfig delete do-not-reject-pages-based-on-content-type
