How AppResponse Selects Top Elements
Each appliance has a finite amount of disk space for storing detailed metric data. To ensure that the most useful and relevant data is retained, AppResponse applies a set of criteria to select the top network elements—such as IP addresses, IP conversations, and Business Groups—for each time window, and retains detailed metric data for these elements only.
General Notes
When selecting the top elements for a specific time window, AppResponse considers criteria such as:
Bidirectional Throughput (top talkers)
Preferred IPs
IP addresses that triggered Static-Any alerts
Business Group membership (to ensure that all Business Groups are included)
The topping criteria for each type of element are discussed in more detail in Topping Criteria by Group/Drilldown Categories.
Selecting Top Elements vs. Calculating Total Traffic Metrics
Keep in mind that, while AppResponse retains detailed metrics for top elements only, it calculates total traffic, throughput, and utilization metrics based on ALL traffic seen by the appliance. For example, suppose an appliance sees traffic for 20,000 IP addresses over a one-minute time window. AppResponse will save detailed metrics for the top 5,000 addresses only, but it will calculate metrics for these addresses based on all traffic seen by the appliance.
Processing Limits
AppResponse can process up to
300,000 unique IP addresses per minute
1,000,000 unique IP conversations per minute
In this context, a “unique IP conversation” is a conversation between two tiers that involves one and only one Defined Application.

AppResponse uses the following criteria to determine the top IP Addresses and Member IPs for each time period:
Throughput—The IP addresses with the highest bidirectional throughput for that time period
Preferred IPs—IP addresses specified in the Preferred IP Manager are included, even if they are not top talkers.
Static-Any alerts—IP addresses that triggered Static-Any alerts during the time window are included, even if they are not top talkers.
Business Groups—AppResponse includes at least one Member IP for each Business Group, even if it is not a top talker for that time window.
Each Member IP included on this basis—not as a top talker, but to ensure that all Business Groups are included—is stored once, even if it is a member of multiple Business Groups.

AppResponse uses the following criteria to determine the top Connected IPs for each time period:
Throughput—The IP addresses with the highest bidirectional throughput (IP address<—>Business Group) for that time period
Preferred IPs—IP addresses specified in the Preferred IP Manager are always included, even if they are not top talkers.
Business Groups—AppResponse includes at least one Connected IP for each Business Group, even if it is not a top talker for that time window.
Business Groups, Applications—AppResponse includes at least one Connected IP for each Business Group and each User-Defined Application, even if it is not a top talker for the time window.
An IP address can be stored multiple times as a Connected IP if it talks to multiple Business Groups.
IP Conversations
In this context, an IP Conversation is an exchange of traffic between two IP addresses that involves one and only one defined application. (A defined application is an application with an entry in the Defined Application Manager.) If address A and address B exchange traffic for three different defined applications, this will result in three different IP conversations.

AppResponse uses the following algorithm to determine the top IP Conversations for each time window:
1)
Divide all conversations in that time window into two sets:
a)
Conversations that include Preferred IPs
b)
Conversations that do not include Preferred IPs
If set a) is greater than or equal to 4,000, select the conversations from set a) using the algorithm described in step 2).
If set a) is less than 4,000, include all conversations from set a) and then select the remaining conversations from set b) using the algorithm described in step 2).
2)
Select the top conversations from set a) or set b) using the following algorithm:
a)
Rank all conversations in the set by bidirectional throughput.
b)
For the highest-throughput address pair, add all conversations (one per conversation per defined application) to the Top IP Conversations list.
c)
Go to the next-highest-throughput address pair and repeat step b).
d)
Repeat step b) and step c), iterating through the highest-throughput address pairs until the Top IP Conversations list is full (4,000 conversations for the time window).
Business Groups
AppResponse does not limit the number of Business Groups that can be stored per time period. It does apply the following rules when storing Business Group data:
When calculating traffic and performance metrics for a Business Group, AppResponse considers all traffic (all applications and all IP addresses) seen by the appliance in that time window.
AppResponse stores all Defined Applications seen in each Business Group.
AppResponse stores the top 20 Other Applications (based on bidirectional throughput) in each Business Group.

NetFlow Data Topping
An appliance can store up to 50,000 unique IP conversations for time window. In this context, a "unique IP conversation" is a conversation in which the two IPs, the Defined Application, and the interface are unique. Topping criteria is selected based on bidirectional throughput.
The 50,000 maximum is divided by the number of interfaces that the appliance monitors in a specific time window. For example:
An appliance collects data from 50 interfaces in a specific minute; for that minute, it stores the top 1,000 unique conversations for each interface.
An appliance collects data from 100 interfaces in a specific day; the appliance aggregates the data from those 100 interfaces to 1-day granularity; for that day, it stores the top 500 conversations for each interface.
