AppNetwork Path Analysis Insight
The AppNetwork Path Analysis Insight is useful for analyzing a specific end-to-end network path. You can use this Insight to
View a topology of the hopes and paths between the two IPs, and to see the paths for which hop-delay metrics are—and are not—available.
Identify devices and interfaces (including Layer-2 devices) in the path that have less-than-optimal performance.
Compare the performance of individual hops in the path.
This Insight groups SNMP data into tables (one row per hop and, where applicable, per interface) so you can compare the performance at each hop. You can compare hops based on criteria such as round-trip times between hops and per-hop/per-interface performance.
Note: Although the data displayed in this Insight is historical and corresponds to the project time, the application path is determined for the current network. Thus, it should be considered a possible path and not necessarily the exact path at the time of interest.

Prerequisites
For best results, it is good practice to do the following:
Let NetSensor collect data in AppPath Analysis mode for at least one hour. This will ensure that it has had enough time to collect a sufficient set of network metrics.
Let NetSensor run overnight after discovering new devices. This lets it do a better job of inferring links for Layer 2 devices in the path.
Metrics
This insight displays two different types of metrics:
Hop Delays are calculated once, when the path was calculated, and are not affected by the project time.
Hop Delay metrics are computed as follows:
–
Hop Delays are not calculated for any devices/systems for which RTT Samples are not reliable. Typically layer-two devices that were inferred to be in the path do not have reliable RTT Samples.
–
To gather RTT samples, NetSensor connects to the device closest to the end point and uses the standard traceroute tool on the node.
–
The insight calculates the mean of the samples for a hop and then subtracts this mean from the previous hop for which RTT Samples are reliable. This is used as the Hop Delay between the current and the previous hop.
–
RTT Samples are considered reliable if both of the following conditions are true:
The previous hop is an end point, or is one of the points from which the traceroute was issued.
The samples do not appear to have been artificially delayed, thereby causing a negative value as a result of the subtraction.
The insight imports sampled metrics from NetSensor, typically through SNMP polling. Note the following:
–
Unlike Hop Delay metrics, sampled metrics are based on the current project time. Therefore, these two sets of metrics might not cover the same time period.
–
Threshold violations for sampled metrics are based on the peak value for each metric within the current project time. The peak value is
the maximum of all raw samples (if the project time is recent and fairly short),
or
the maximum of all aggregated samples (if the project time is older or spans multiple days).
Refer to the NetSensor documentation for descriptions of sampled metrics (such as Interface Incoming/Outgoing Utilization, CPU Utilization, and so on).
Metric Thresholds
Each metric has a Warning and a Critical threshold setting. The insight colors objects and thresholds as follows:
Green = The metric is below the Warning threshold.
Yellow = The metric is equal to the Warning threshold, but still less than the Critical threshold.
Red = The metric is equal to or greater than the Critical threshold.
Gray = There is no metric data available for the object.
If a hop has multiple threshold settings defined, the icon is red (if any threshold is critical) or yellow (if one threshold is a warning and none are critical).
You can change thresholds after the path is generated. The insight then re-applies all thresholds and redraws all screens with the appropriate colors.
Workflow Description
The following steps outline the general workflow for this Insight.
1)
Identify an IP Conversation whose network path you want to analyze. IP conversations with high rates of packet loss and/or retransmissions are often good candidates for investigation.
2)
In the insight, chart, or table window, right-click on the IP conversation of interest and choose Analyze In > Element Polling > Application Path Analysis. This allows an end-to-end workflow in which you
a)
Search for problematic IP conversations in the Desktop Console.
b)
Send the end points to the AppNetwork Path Analysis Insight.
If you already know the end-point IPs of the path you want to investigate, you can open the insight directly (Insights > Element Polling > AppNetwork Path Analysis); when the insight opens, click the New Button in the top-left corner and enter the two IPs manually.
➥ When APM Element Insight receives the two end points, it polls the network and determines the path between the two points.
Note—Path analysis could take several minutes if there are problems with login credentials for devices in the path.
➥ After NetSensor determines a network path:
• The Forward Path and Return Path hop lists are populated.
• Links to various SNMP Comparison views that might be useful are added under the Additional Insights tab.
• Information about devices on the path is added under the System Details tab.
3)
Analyze the network path using the panes and tabbed pages:
–
Topology Pane (Top)
–
Path Summary
–
Path Delay Analysis
–
Path System/Device Types
–
Path Generation Details
–
Threshold Settings Tab
If you see unexpected results, see Troubleshooting this Insight.
Topology Pane (Top)
The topology pane uses the following conventions to visualize the network path:
NetSensor computes two paths for each request: the forward path (source ==> destination) and the return path (destination ==> source). The arrows in the topology visualize the links between the two IPs.
The insight colors each hop and link based on its threshold settings and the metric calculated for the current project time. To view metric information in a tooltip, rest the cursor over the hop or link.
Links are displayed as follows:
–
Solid blue line = a link for which there is no Hop Delay metric available
–
Dotted line = multiple links for which a Hop Delay was calculated. The end points of the dotted line represent the two hops used to calculate the delay; for more information, see Metric Thresholds.
A dotted line (hop delay calculated) can span several solid lines (no hop delay calculated) when hop-delay metrics are not available for some links in a path.

Path Summary
The metrics for the devices/systems and links are compared to the Threshold settings and compliance ratios displayed in this tab. For details about each metric, click View. The insight either switches to a different tabbed page or opens a secondary insight to compare the metric thresholds.
To examine possible correlations between performance at a specific hop or interface vs. performance along the entire path, click the View button next to System Performance or Interface Performance. This opens a corresponding secondary Insight that shows metrics for all observed hops and interfaces in the network path. To compare hop-versus-path performance:
1)
In the Hop List, select a metric cell (or use Ctrl-click to select multiple metric cells).
➥ The hop time-series chart (center) shows data for the selected hops.
2)
Compare the hop data to the network time-series chart (bottom). By default, this chart shows Response Time Composition for the network, which can help you correlate delays to specific hops in the path. To compare hop data against a specific AppResponse metric, click the Comparison to Reference Metric tab and choose the metric you are interested in.
Note—The secondary Insights treat the source of the application path as a server and the destination as a client. This might produce some odd metrics if the source is actually a client.

Path Delay Analysis
Shows the forward and return paths in tabular format. The column headers are:
Hop — The relative hop number for this path
System Name – Primary name of this device/system as managed within NetSensor
Interface – Outbound interface used on this node
Hop Delay – Cell color is based on the defined threshold. For more information, see Metric Thresholds.
RTT Samples – Cell color is based on the defined threshold. For more information, see Metric Thresholds.
Path System/Device Types
Shows the summary of what each device/system type on the path. The column headers are:
Forward Hop – Relative hop number on the forward path. If no value is displayed, the device or system did not participate in the forward path.
Return Hop – Relative hop number on the return path. If no value is displayed, the device or system did not participate in the return path.
System Name – Primary name of this device/system as managed within NetSensor
Uptime – Device uptime at the end of the current project time (“To” field in the Desktop Console)
Location – Polled from the system/device MIB
Description – Polled from the system/device MIB
Path Generation Details
This page displays a summary of how the path was generated. This information is useful when the path could not be generated.
Threshold Settings Tab
It shows the currently in effect threshold settings. If these are changed, the change is a local operation applied to this generated path only. It is not a global change that affects all paths.
For more information, see Metrics and Metric Thresholds.
Troubleshooting this Insight
Problem:
The AppNetwork Path Analysis Insight is having trouble obtaining an application path.
Solution:
Do the following:
Make sure that NetSensor is configured with authentication and login information for all devices in the network that might be part of the path.
