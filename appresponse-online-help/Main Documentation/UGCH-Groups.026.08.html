Monitoring Interface Groups
Many networks are divided into logical segments by firewalls, switches, load balancers, and other devices. To analyze and troubleshoot these of types of devices effectively—to validate traffic policies enforced by a specific device, for example—it is often necessary to observe traffic in multiple locations and then to consider these locations when analyzing traffic.
A monitoring interface group (MIFG) is a logical grouping of one or more physical interfaces. AppResponse treats all physical interfaces in the same group as one interface. Users can define groups that correspond to specific network segments (for example, “local_LAN” or “firewall_WAN_side”).
AppResponse can group monitored traffic by the MIFG in which the traffic was observed. You can then view traffic by MIFG in the Desktop Console. This functionality is useful for workflows where you need to consider the network segment where traffic was observed, such as
Isolating traffic to specific links (such as WAN links)
Validating and analyzing traffic policies across different network segments
Viewing and comparing IP conversations across multiple tiers

Configuring Monitoring Interface Groups
You define and configure your Monitoring Interface Groups in the System > Interface Groups page in the System > Administration web UI (Figure 25). The default configuration consists of one group defined for each monitoring interface on the appliance (MIFG_1, ...MIFG_n).
Note the following:
You must assign every interface on an appliance to one and only one MIFG.
It is good practice to configure your MIFGs so that each IP conversation pair of interest is observed in one interface group only (not in multiple groups).
If you have multiple interface groups that see the same IP conversation pairs, configure your groups so that only one group is included in the total traffic metrics. To do this, select the Limit Reporting to the Monitoring Interface Group checkbox for the other groups.
If the MiFG data collection is enabled, you might see inaccurate metrics for a TCP connection if
–
The connection transfers packets across multiple MiFGs, and
–
The appliance does not observe all packets in any one MiFG—that is, packets for that connection are split across multiple MiFGs.
For more information, see NOTE: If Packets for the Same Connection are Observed in Different MiFGs, Some Traffic Might Be Ignored.
If you are configuring a appliance also supports High Speed Captures (such as a 4100), the MIFGs you define in this page are used for both captured packets and for monitored traffic.

Traffic Direction Relative To
This option determines how AppResponse determines Inbound vs. Outbound traffic relative to the MIFG. You can specify one of three options:
Internal Addresses—Traffic direction is relative to the Internal Address List specified in the Setup page of the System > Administration web UI.
Custom Addresses—This option enables you to specify an alternate set of internal addresses that apply to Monitoring Interface Groups only.
Tap—This option is useful when a Monitoring Interface Group includes two interfaces that are connected to unidirectional tap ports (instead of a span port). AppResponse automatically assigns the Inbound direction to the lower-number port and the Outbound direction to the higher-number port.
NOTE—This option is available only when exactly two interfaces are included in a Monitoring Interface Group.
Limit Reporting to the Monitoring Interface Group
Traffic observed on each AppResponse monitoring interface is normally reported in all tables. Enable this option for a Monitoring Interface Group to exclude reporting of observed traffic—that is, traffic observed on any of the group's monitoring interfaces—from all tables except the Mon Interface table (View > Tables > Mon Interface Group).
This option is useful in the following situations:
You want to exclude traffic observed in some network locations from your overall traffic results.
You have multiple MIFGs that can observe the same IP conversations.
AppResponse de-duplicates traffic from multiple MIFGs in all tables except the Mon Interface Group table. This de-duplication is based on one-minute averages of IP conversations, not on individual packets. If AppResponse observes the same conversation over one minute, but the traffic occurs at different times within that minute, the resulting de-duplication results will be inaccurate. For this reason, AppResponse cannot guarantee accurate results from the de-duplication of this type of data (one-minute averages of the same IP conversations).
To avoid this possible scenario, it is good practice to enable this option for all MIFGs except those that observe completely unique IP conversations (no conversations observed by that MIFG can be observed by any other MIFG).
The appliance observes the "same" IP conversations from multiple MIFGs, but these conversations represent distinct and separate traffic flows.
For example, suppose an appliance observes traffic on opposites sides of a transparent proxy such as a WAN accelerator. Instead of simply transferring traffic between the WAN and LAN, the device might compress, cache, pre-fetch, re-write, and otherwise transform traffic so that the "same" IP conversation (client<==>server) is actually two separate and distinct conversations (client<==>proxy and proxy<==>server). In this case, you might want to avoid combining the LAN and WAN conversations and include only one set of conversations in your overall traffic results.
NOTE: If Packets for the Same Connection are Observed in Different MiFGs, Some Traffic Might Be Ignored
This section describes how data collection for Monitoring Interface Groups can affect the metric results for individual connections. In this context, a connection is the set of packets exchanged between two hosts over the same TCP port.
If the Collect data for Monitoring Interface Groups option is enabled, you might see inaccurate metrics for a connection if
The connection transfers packets across multiple MiFGs, and
The appliance does not observe all packets in any one MiFG—that is, packets for that connection are split across multiple MiFGs.
Key Concept—With MiFG data collection enabled, metrics for each connection are based only on packets in the MiFG where the most packets were observed. Packets observed in other MiFGs are ignored.
When an appliance observes packets for the same connection in different MiFGs, it calculates metrics for that connection as follows:
MiFG data collection off:
The appliance
–
Discards duplicate packets (==> the same packet observed in multiple MiFGs).
–
Calculates metrics based on all unique packets observed for each connection (regardless of the MiFG in which a particular packet was observed).
MiFG data collection on:
The appliance
–
Considers the packets for the “highest-packet” MiFG (that is, the MiFG in which the most packets for an individual connection were observed).
–
Ignores all other packets for that connection that were observed in other MiFG.
–
Calculates metrics based only on the packets in the highest-packet MiFG.
This is not an issue if the appliance sees all packets for a connection in one MiFG or another. It can be an issue if some packets are observed only in a MiFG that is not the highest-packet MiFG.
The most common real-world example of when this can occur is when a bidirectional connection between two hosts uses separate unidirectional flows that are observed in different MiFGs. In Figure 21, the appliance observes 300 packets for a connection: 100 packets (A --> B) in one MiFG and 200 packets (A <-- B) in another.

If MiFG data collection is turned off, the appliance considers the packets from both MiFGs and collects metrics based on all 300 packets.
