“Individual Page Views” Insight
This insight is useful for identifying and analyzing page views of interest. You can find top page views based on performance or utilization metrics such as page download times; percentage of slow pages; or percentage of 3xx, 4xx, or 5xx response codes. You can also find the top page views for selected IPs, applications, page families, or users. If the appliance has captured packets for an individual page view, you can download these packets for in-depth analysis in Transaction Analyzer or other programs. This insight is available on both appliances and Directors.
Figure 1
“Individual Page Views” Insight
 
To use this insight, do the following:
Procedure 1
Using the Individual Page Views Insight
1.
Open the Web Applications table (Desktop Console > View > Tables > Web Applications...).
2.
Drill down to and select one of the following:
• 
Client IP
• 
Originating IP
• 
Page Family
• 
Server IP
• 
User
• 
Web Application
3.
Right-click on the selected item and choose Send To > Individual Page Views.
The Individual Page Views insight appears. Note that
• 
The item you right-clicked on is selected in the Group/Drilldown Chooser (top left).
• 
The time series graph (top right) shows a summary graph of all page views for the selected item within the current project time window.
• 
The Top Individual Page Views table (bottom) shows the top calls for the selected item within the selected project time window.
• 
The metric and topping criteria in the graph and table are based on the Top: and By: fields in the top-left corner of the insight window.
4.
At this point, you can do the following:
• 
Select a different IP, User, Web Application, or Page Family:
The three tabbed panes provide different interfaces for finding and selecting an item of interest. When you select an item in one of these panes, the graph and table updated automatically.
• 
Select another topping threshold or performance metric for the selected item:
Set the Top: and By: fields. The time series graph and table update automatically.
• 
Change the metrics shown in the Top Page Analysis Groups table (top left):
• 
Informational
• 
Page Performance and Page Utilization (see Page Performance Metric Definitions)
• 
Change the metrics shown in the Top Individual Page Views table (see Individual Page View Metric Descriptions).
• 
Drill down from a page view to the component resources in each view.
The Waterfall graph (Figure 2) shows the Server Busy Time for individual resources and for the entire page. This view is useful when you want to identify when server processing delays contribute significantly slow page times.
To open a Waterfall graph, right-click on a page view in the Individual Page Views table (bottom) and choose View Waterfall (Resource Timeline). A Waterfall graph enables you to see clearly the server processing delays (pink) for individual resources, as well as the intervals when the server processes multiple requests in parallel.
Figure 2
“Individual Page Views Waterfall (Resource Timeline)” Graph
 
• 
Download packets for an individual page view:
In the Top Individual Page Views table, right-click on a row and choose one of the following:
• 
Analyze in Transaction Analyzer Tree View
• 
Analyze in Transaction Analyzer AppDoctor
When you save in Transaction Analyzer, Transaction Analyzer saves both the trace file (<filename>@<timestamp>@<appliance>.appcapture) and the Transaction Analyzer file (<filename>.atc.m) in the same directory.
• 
Analyze in Transaction Analyzer Trace Explorer
When you save in Trace Explorer, Transaction Analyzer saves the downloaded packets in a trace file (<filename>@<timestamp>@<appliance>.appcapture).
• 
Send Packets to Wireshark
• 
Download Packets
• 
Open Page in Web Browser
Note—These operations are supported only if a packet capture was running at the time that the specific page view was observed by the AppResponse appliance.
Note—Transaction Analyzer does not retain downloaded packets on the local host unless you save the resulting file in Trace Explorer or Transaction Analyzer. If you close either window without saving, the downloaded packets will be discarded on the local host. (If you want to view these packets again, you will need to download them again before they are overwritten in the appliance buffer.)
End of Procedure 1

HTTP Status
An individual page view typically consists of multiple objects: one main object (such as an HTML page) and one or more constituent objects (the images, scripts, style sheets, and other objects included in the page). The HTTP Status is the HTTP response code returned by the server for the main object.
If you want to view the exact HTTP response codes for individual objects in a page view, download the packets and examine the protocol decodes in Transaction Analyzer.
For information about specific response codes, go to the World Wide Web Consortium website (www.w3.org) and search on the string “http status code definitions”.
HTTP 100s
The total number of HTTP responses (100-199) for the main object and all constituent objects in the page view.
HTTP 200s
The total number of HTTP responses (200-299) for the main object and all constituent objects in the page view.
HTTP 300s
The total number of HTTP responses (300-399) for the main object and all constituent objects in the page view.
HTTP 400s
The total number of HTTP responses (400-499) for the main object and all constituent objects in the page view.
HTTP 500s
The total number of HTTP responses (500-599) for the main object and all constituent objects in the page view.
HTTP Responses
The total number of HTTP responses for the main object and all constituent objects in the page view.
Network Busy Time (per resource)
Network transfer time for the full payload of an individual request or response, from the first observed packet to the last.
Network Busy Time (per page)
Network transfer time for all resources in a page. Because the server might have processed multiple requests in parallel, the total-page-view busy time might be less than the sum of the busy times for all resources in the view.
Page Time
 
Page Traffic [MB]
 
Request Traffic [MB]
 
Referrer
The web page from which the user requested the page view (usually by clicking a link on the referrer page).
Server Busy Time (per resource)
The time that the HTTP server spent processing an individual resource request. This time is measured from the resource request (last observed packet of HTTP request) to the initial response (first observed packet).
Server Busy Time (per page)
The total time that the HTTP server spent processing all resource requests in a page. Because the server might have processed multiple requests in parallel, the total-page-view busy time might be less than the sum of the busy times for all resources in the view.
User agent
The web client software used to generate the page request.
