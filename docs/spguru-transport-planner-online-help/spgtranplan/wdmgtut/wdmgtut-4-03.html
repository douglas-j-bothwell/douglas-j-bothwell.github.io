Routing Automatically

The examples in this section show routing actions at the OCH and DCL layer. The aim of routing is to establish as much traffic as possible within the given capacity of a network layer. These examples describe the impact of the main routing options. For more information on routing options, see the SP Guru Transport Planner User Guide.

Procedure 3-2 Routing Traffic in the Optical Layer (OCH Layer)

  1. Open the WDMGuru_Examples project.
    1. Select File > Open....
    2. Select the WDMGuru_Examples project, then press Open.

      3. The example project, which contains multiple scenarios, is loaded. The Australia scenario appears in the workspace.

  2. Switch to the USA scenario.
    1. Select Scenarios > Switch To Scenario > USA.

      2. The USA scenario appears in the workspace.

    2. Use the OMS button on the toolbar to inspect the OMS layer. Note that initially one fiber pair is lit on each link.
    3. Use the OCH button on the toolbar to inspect the OCH layer. Note that no wavelengths are in use on the links.
  3. Route an OCH traffic matrix (with node limitations).
    1. Click the button on the toolbar or select the Design > Route DCL/OCH Traffic dialog box.
    2. Select OCH Layer as the Layer and OC-48 as the Traffic Matrix.
    3. Choose the following options:
      • Unprotected (Protection)
      • Sequential (Routing Algorithm)
      • Unsorted (Connection Order)
      • Hop Count (Routing Cost)
      • Impose Node Limitations (Node Model Options)
    4. Uncheck Allow Equipping Dark Bands in the Link Model Options panel. For more information on these options, see the SP Guru Transport Planner User Guide.



    5. Press Route.

      6. The Routing Results dialog box appears.



      Only 35 percent of the traffic could be accommodated in the network. Fourteen percent of the link capacity in the OCH layer is in use and 84 percent of the OXC capacity. Notice that the node utilization is much higher than the link utilization, and therefore the capacity shortage must be in the nodes. Also, some other statistics are reported, such as the average hop length of the connections. Access the Routing Results dialog box at any time using Design > Routing Results.

    6. Close the Routing Results and the Route DCL/OCH Traffic dialog boxes.
  4. Inspect the routes of the connections.
    1. Select Network > Connection Browser.
    2. Select OCH for Traffic, All for Layout, and Traffic Matrix for Order.
    3. Click on the + icon (next to OC-48) to inspect the connection details of traffic matrix OC-48.

      4. For each connection, the following information is shown:

      • its capacity (Capacity)
      • whether it is routed (Routed)
      • whether it is carried on mesh or on rings (Carrier)
      • its protection type (Protection)
      • its number of hops (Hops), length (Length), delay (Delay), etc.

        • When you select a connection in the browser, its route is visualized in the workspace. Click on the + icon (next to a connection), to see the route of the connection in the connection browser. To inspect the wavelengths used on the links by the selected connection, mark Show Wavelength Configurations in the lower-left corner of the connection browser.



    4. Close the connection browser.
    5. Select Info > Topology > OCH Layer. This dialog box provides additional network-wide statistics for the OCH layer, such as the node degree and the node and link utilization. Similar reports can be generated for each of the other layers.



    6. Close the OCH Layer Topology dialog box.
  5. Tear down the OCH traffic matrix.
    1. Select Design > Tear Down Traffic....
    2. Select the OCH Layer and the (partly accommodated) traffic matrix OC-48, then press Tear Down.

      3. The OC-48 traffic matrix is released from the network and all capacity in the OCH layer becomes available again.



    3. Close the Tear Down Traffic dialog box.
  6. Route an OCH traffic matrix (without node limitations).
    1. Select the Design > Route DCL/OCH Traffic... dialog box, choose OCH Layer as the Layer, and OC-48 as the Traffic Matrix.
    2. Choose the following options:
      • Unprotected (Protection)
      • Sequential (Routing Algorithm)
      • Unsorted (Connection Order)
      • Hop Count (under Routing Options, next to Routing Cost)
      • Uncheck Allow Equipping Dark Bands in the Link Model Options panel, and also the Impose Node Limitations option.
    3. Press Route.

      4. The Routing Results dialog box appears.

      Observe that 100 percent of the traffic is routed. Because the node limitations were not imposed, the discrete nodes have been upgraded to the correct size to allow them to route the entire traffic matrix.



  7. Close the Routing Results and the Route DCL/OCH Traffic dialog boxes.
  8. To inspect the nodes in the OCH layer more in depth, open the Network > Node Browser, and then select Ports as the View and OCH as the Layer.



  9. Close the node browser.
  10. Tear down the OCH traffic matrix.
    1. Select Design > Tear Down Traffic....
    2. Select the OCH Layer and the (entirely accommodated) traffic matrix OC-48, then press Tear Down.
    3. Close the dialog box.
  11. Route an OCH traffic matrix (with protection).
    1. Select Design > Route DCL/OCH Traffic.... In the dialog box, select OCH Layer and the traffic matrix OC-48.
    2. Choose the following options:
      • Protected (Protection)
      • Node Disjoint
      • OTS Link Disjoint
      • Sequential (Routing Algorithm)
      • Unsorted (Connection Order)
      • Hop Count (Routing Cost)
      • Uncheck Client Protection
      • Uncheck Allow Equipping Dark Bands (Link Model Options)
      • Uncheck Impose Node Limitations (Node Model Options)



    3. Press Route.

      4. The Routing Results dialog box indicates that the traffic matrix OC-48 could not be entirely accommodated in the network. Therefore, there must be a capacity shortage on the links (as indicated by the high link utilization), because not all traffic can be assigned a working path and a diverse protection path.



    4. Close the Routing Results dialog box.

      5. Note that the traffic matrix OC-48 remains part of the list of traffic matrices in the Routing dialog box, because it is not entirely routed. This gives the opportunity to try to route the unrouted part of the traffic matrix, for example, after adding link or node capacity.

    5. Close the Route DCL/OCH Traffic dialog box.
  12. Inspect the routes of the connections.
    1. Select Network > Connection Browser.
    2. Select OCH for the Traffic, All for the Layout, and Traffic Matrix for the Order.
    3. Click on the + icon (next to OC-48) to inspect the connection details of traffic matrix OC-48. Select a connection: the working path (in green) and back-up path (in red) are shown on the workspace.



    4. Close the connection browser.
  13. Close the project
    1. Select File > Close.
    2. Select Don't Save in the Close Confirm dialog box.

End of Procedure 3-2

Procedure 3-3 Routing Traffic in the SONET/ SDH Layer (DCL Layer)

  1. Open the WDMGuru_Examples project.
    1. Select File > Open....
    2. Select the WDMGuru_Examples project an press Open.

      3. The example project, which contains multiple scenarios, is loaded. The Australia scenario appears in the workspace.

  2. Select Scenarios > Switch To Scenario > USA to view the USA scenario in the workspace.
  3. Route an OCH traffic matrix (without node limitations).
    1. Select Design > Route DCL/OCH Traffic... dialog box.
    2. Select OCH Layer, and the traffic matrix OC-48.
    3. Choose the following options:
      • Unprotected (Protection)
      • Sequential (Routing Algorithm)
      • Unsorted (Connection Order)
      • Hop Count (Routing Cost)
      • Uncheck Allow Equipping Dark Bands (Link Model Options)
      • Uncheck Impose Node Limitations (Node Model Options)
    4. Press Route.

      5. The Routing Results dialog box appears.

      The traffic matrix OC-48 is entirely accommodated in the network.



    5. Close the Routing Results and Route DCL/OCH Traffic dialog boxes.
    6. Switch to the DCL layer view using the DCL button in the toolbar.

      7. Note that the traffic routed in the OCH layer results in a logical (or virtual) topology in the DCL layer. This is because each non-native connection that is routed in the OCH layer results in a logical DCL link with capacity corresponding to the bit rate of the OCH connection. DCL traffic can be routed on these logical DCL links (see the next step).

  4. Route a SONET/SDH traffic matrix.
    1. Select the Design > Route DCL/OCH Traffic... dialog box.
    2. Select DCL Layer, and the traffic matrix STS-1.
    3. Choose the following options:
      • Unprotected (Protection)
      • Sequential (Routing Algorithm)
      • Longest First (Connection Order)
      • Hop Count (Routing Cost)

        • Note that the option Impose Node Limitations is always checked and cannot be unchecked for DCL traffic, because continuous node models are being used in the DCL layer (this can be verified in the Network > Network Properties dialog box).

    4. Press Route.

      5. The Routing Results dialog box appears.

      The traffic matrix STS-1 is entirely accommodated in the network.



    5. Close the Routing Results and Route DCL/OCH Traffic... dialog boxes.
    6. Switch to the DCL layer using the DCL button in the toolbar. Note that some link capacity at the DCL layer is in use now.
  5. Inspect the routes of the SONET/SDH connections.
    1. Select Network > Connection Browser.
    2. Select DCL for the Traffic, All for the Layout, and Traffic Matrix for the Order.
    3. Click on the + icon (next to STS-1) to inspect the connection details of traffic matrix STS-1.



    4. Close the connection browser.
  6. Tear down the traffic matrices.
    1. Select Design > Tear Down Traffic....
    2. Select the DCL Layer and the traffic matrix STS-1, and press Tear Down.

      3. The DCL traffic matrix is released from the network, and all capacity in the DCL layer becomes available. The OCH traffic matrix OC-48 is still accommodated in the network.

    3. Select the OCH Layer; select OC-48 and press Tear Down.

      4. The OCH traffic matrix is released from the network. Note that the order in which traffic matrices are torn down is important. Before tearing down an OCH traffic matrix, any dependent DCL traffic matrix must be torn down first. If you do try to tear down an OCH traffic matrix that is still supporting routed DCL layer traffic, you will receive an error message directing you to first perform a tear down operation for DCL traffic matrices.

    4. Close the Tear Down Traffic dialog box.
  7. Close the project
    1. Select File > Close.
    2. Select Don't Save in the Close Confirm dialog box.

End of Procedure 3-3
