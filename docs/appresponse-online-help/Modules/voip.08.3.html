Deployment Considerations
Many factors must be considered when choosing where to install the AppResponse Appliance within your network. When choosing a target location for AppResponse Appliance, take the items in the following sections into consideration. The Module for VoIP Performance is intended to monitor end-user calls originating or terminating at a VoIP handset or “soft phone”. Because there are numerous methodologies for hybrid VoIP deployments, consult your Riverbed sales representative to determine if your environment is appropriate for deployment of the Module for VoIP Performance.
Network Coverage
The location of an AppResponse Appliance dictates the types of network traffic it monitors. In general, when used for monitoring VoIP, the AppResponse Appliance is best installed at a traffic aggregation point in order to maximize the amount of VoIP network traffic. Ideal aggregation points include the backbone/workgroup Ethernet switch or tapped into a network segment where both media (RTP Stream) and VoIP Signaling are present.
Span/Mirror Port Monitoring Interface Configurations
If using the span/mirror port monitoring interface configuration, AppResponse Appliance must be attached directly to the device, typically a backbone/workgroup Ethernet switch that supports span/mirroring features. Because each network equipment vendor implements the span/mirror port features differently, some network equipment might have limitations in its use with AppResponse Appliance. In general, however, all major switch vendors support basic traffic mirroring functionality.
Taps
Refer to the AppResponse Operations Guide or Hardware Installation Guide.
Modified Frame Formats
Some networks use modified layer 2 and layer 3 frames to facilitate VLAN definitions and prioritized traffic flow control. AppResponse Appliance can properly operate in environments where 802.1q VLAN tagging and 802.1p priority tagging are used. AppResponse Appliance does not work in environments where other modified frame formats, such as ISL format or Ethernet jumbo frames, are used.
Encryption, Tunneling, and Encapsulation
AppResponse Appliance collects metrics by extracting information from VoIP signaling and RTP packets. Many network technologies modify the IP frame format through encryption or introduction of additional encapsulation headers. These headers can impair AppResponse Appliance’s ability to generate meaningful quality metrics. For example:
Layer 3 header obscurity (MPLS, PPPoE, etc.)—These protocols insert an additional header before OSI layer 3 (Network Layer – IP protocol). As a result, the headers of the IP protocol (layer 3) and all layers above are not recognized by AppResponse Appliance. It is suggested that an installation location be selected that reduces or eliminates traffic employing layer 3 header obscurity.
Layer 4 header obscurity (IPSEC, GRE, GTP, etc.)—These protocols insert an additional header before OSI layer 4 (Transport Layer – TCP/UDP protocol). As a result, the layer 4 protocol is not recognized, impairing AppResponse Appliance’s ability to generate meaningful information for VoIP calls appearing in the network. It is suggested that an installation location be selected that reduces or eliminates traffic employing layer 4 header obscurity.
Obscurity above layer 4 (SSL, TLS, SHTTP, SRTP, etc.)—These protocols encrypt data above the transport layer and can impact AppResponse Appliance’s ability to collect and generate meaningful VoIP quality metrics. In environments where Secure RTP (SRTP) is being used, it is suggested that only the RTP payloads be encrypted and not the RTP headers or RTCP packets. For SSL, SSLv2 and SSLv3 are supported. For TLS, versions 1.0, 1.1, and 1.2 are supported.
Network Address Translation
The AppResponse Appliance identifies the source and destination for each packet based on the addresses in the packet IP header. Network address translation (NAT) is a procedure that is common in firewalls, load balancers, and proxies. NAT replaces the original endpoint IP address with the IP address of an intermediary device, usually the firewall or other device performing the address translation. As a result, all network sessions appear to originate from the translated network address rather than the actual originating device. Because the original endpoint IP address is useful during VoIP performance analysis and fault diagnosis, it is best if AppResponse Appliance is installed at a network location before the NAT process.
Security
Monitoring interfaces operate in promiscuous mode to record traffic only. These interfaces are not assigned IP addresses and are not able to transmit network traffic. Because the monitoring interfaces are not assigned IP addresses, it is safe to connect these interfaces to a network segment outside a firewall or directly to the Internet.
The AppResponse Console is used for general-purpose communications and should be connected to a network segment protected against direct access from the “outside world”.
