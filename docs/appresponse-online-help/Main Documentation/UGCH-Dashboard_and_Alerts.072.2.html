Alert Manager
The alert manager is used to create, edit and delete static alerts, and to customize adaptive alerts. To begin managing alerts, click the Desktop Console Tools menu, then select Alert Manager.
Figure 1
Alert Manager
 
The Alert Manager comprises three sections: the Alert Manager Information Pane at the top; the tabbed work area, where configuration takes place, as reviewed below; and the Control Bar at the bottom.
A newly installed appliance does not have any static alerts defined. However, it lists a number of adaptive alerts. The alert manager can be used to modify adaptive alerts (see Customizing Adaptive Alerts).
If your appliance belongs to a domain, you should review distributed alerts. (See Domains and Distributed Alerts)
Alert Manager Information Pane
The Alert Manager Information Pane displays various parameters of an alert in tabular form. Most of these are specified using the tabbed work area: Name, Description, Type, Status, Dashboard Section, Group, Condition, Days, End Time, Frequency, and Actions. These are described in the following sections. Other columns are available:

Edited - a check mark indicates it has been edited and not saved.
Modified By - the last user to edit the alert definition.
Last Modified - the time and data of the last edit.
Current Severity - the current severity of the alert, including Not Occurring.
Right-clicking on a row in the Information Pane displays a context menu for that alert.

Static Alerts
Static alerts are generated when the appliance observes a traffic condition that crosses a user-defined threshold. There are two kinds of static alerts.
Standard Static Alerts (called simply “Static Alerts”) are generated when their conditions, defined as complex Boolean operations on any desired set of metrics, are fulfilled for a single specified group or group path against that group’s total traffic, i.e., for the sum of all applications seen by that group. Static Alerts are useful for monitoring the behavior of a specific group or application; e.g., generate an alert when the Initial Application Response Time for the Business Group SAP Servers Chicago exceeds 100 ms and Connection Request Rate is less than 10/second. You may define up to 100 Static-Alerts per appliance or director.
Static-Any Alerts are generated when their conditions, defined as simple Boolean operations (AND, OR) on up to three metrics, are fulfilled for one or more groups within a specified type of group against either those group’s total traffic (the sum of all applications seen by those groups) or a single application or any of one or more applications within a specified type of application seen by those groups. Applications are divided into two categories for Static-Any Alerts: Defined Applications and Other Applications, e.g., TCP or UDP applications. A Static-Any Alert may not be defined against an undefined non-TCP-non-UDP application.
Note: Because a Static-Any alert with a misconfigured condition (e.g., all IP addresses with Throughput (Inbound and Outbound) > 0) is capable of generating an almost unlimited number of events, only 10 events are reported for each Static-Any alert, regardless of the number of groups that satisfy the alert conditions.
The difference between total traffic and any application is important to understanding Static-Any Condition Types. Total traffic means that the alert condition is evaluated against the sum of all applications seen by that group or set of groups. Any of a set of applications means that the condition is evaluated against each of the applications seen by the group. For instance, a Static Alert set to detect the condition Throughput (Inbound and Outbound > 1 megabit/second against total traffic would trigger if the sum of application activity for the group(s) involved exceeded that threshold. But a Static-Any alert set to detect the condition Throughput (Inbound and Outbound > 1 megabit/second against any application would trigger only if the throughput of any one of the applications specified exceeded that threshold. For details on how such as alert is defined using the Alert Manager, see Group - Static-Any.
Static-Any Alerts are useful for detecting a given behavior on a large set of groups or any groups, e.g., generate an alert when Connections Failed Rate (TCP Servers) exceeds 10/second for any IP address, which would detect any self-propagating worm. You may specify up to 100 Static-Any Alerts per appliance or director.
We first review how to create Static Alerts as an introduction to the use of the Alert Manager, and then review the similar steps needed to create a Static-Any Alert.
Creating a Static Alert
To create a static alert, click the New button in the Alert Manager. The tabs in the Alert Manager work area allow the user to define the identity, group (traffic monitored), condition, schedule, action, and links to Insights for all static alerts. Alternatively, you may right-click an existing alert in the Information Pane and select Duplicate to populate all the tabs with the specifications of that alert, which you may then modify; a unique name is automatically assigned to the new alert.
Identity Tab
The Identity tab is used to configure the alert name, description, type, dashboard section, and status.
Figure 3

This page has the following options:

Dashboard Section—determines in which section of the dashboard to place the notification when the alert is triggered: Client, Network, or Server (see Problem Management Dashboard).

Condition
The Condition tab is used to specify the threshold that triggers the static alert. An alert is generated when the network traffic for the group crosses the specified threshold. For Static Alerts, the Condition tab offers both a basic and advanced interface.
The basic interface lets you specify conditions involving simple Boolean conditions (AND, OR) on up to three metrics.
Figure 6
Alert Manager - Condition Tab
 
Multiple severity levels can be defined in the condition section. The severity is included in the alert notification.
A condition specified in the basic Condition interface may involve a single metric (e.g., server response time > 80 msec), or up to three traffic metrics using a logical expression (e.g., round trip time > 200 msec OR outbound packet loss > 10%).
Click the Advanced checkbox to configure alerts using logical expressions involving more than three metrics. See Advanced Alert Expression Syntax.

Schedule Tab
The Schedule tab is used to configure specific time periods when the static alert is active. By default, the static alert is active at all times, but can be disabled for certain hours of the day or days of the week. The schedule can also be used to specify the Frequency with which the alert condition is checked. Alert conditions can be checked at the following intervals: 1 minute, 5 minutes, 1 hour or 1 day.
Figure 8
Alert Manager - Schedule Tab
 
The Minimum Duration field specifies the minimum duration for which a Static-Any condition must be in continuous violation before the alert is triggered. If you set this field to 10, for example, the alert is not triggered unless the condition is violated continuously for 10 minutes.

Creating a Static-Any Alert
The process of specifying a Static-Any alert is similar to that of a Static Alert. The major difference is in the Group and Condition tabs.
Because a misconfigured Static-Any alert is capable of generating an almost unlimited number of events (e.g., all IP addresses with Throughput (Inbound and Outbound > 0), only 10 events are reported for each Static-Any alert, regardless of the number of groups that satisfy the alert conditions.
Identity - Static-Any
The Identity tab for Static-Any Alerts is the same as for Static Alerts. (See Identity Tab.)
Group - Static-Any
Unlike Standard Static Alerts, Static-Any Alerts allow you to specify a complex set of groups or applications, or in some cases groups and applications, to which the alert applies.

Each list may be specified either as a Set of groups or applications of a single type Including those entered in the text box, or as Any groups or applications of a single type Excluding those entered in the text box, if any.

Evaluating Alert Conditions
After an alert is defined and saved in the database, the alert condition can be compared against the metrics stored in the appliance database. To do so, right-click the alert in the alert manager and select “Evaluate Condition”. The alert condition is evaluated against the metric values for the current time selection.
Figure 12
Alert Manager - Condition Evaluator
 
The condition evaluator displays alert details (name, group, condition and period) and the current value of the metric specified in the alert condition. Other metrics related to the selected metric are also displayed. The Result field at the bottom of the window displays the alert status for the current time selection. If the metric(s) defined in the alert cross the threshold(s) specified in the condition, the Result field indicates that the alert is occurring in this time selection, its priority, and specifically when it would trigger (if its frequency is less than the time selection of the current project).
Adaptive Alerts
The appliance can measure and record a quantity of network data that may overwhelm all but the most advanced network engineers, and even they may not be familiar with some of the correlations and collective behaviors that can reveal the cause and even location of Internet problems.
The Problem Management Dashboard provides a set of built-in analyses called Adaptive Alerts that automatically detect a wide range of problems. These analyses attempt to identify a common problem, such as an overloaded server, by finding outliers in the metrics collected by the appliance. When an outlier is identified, the appliance determines whether this abnormal behavior is correlated with another metric in an attempt to localize the problem. The appliance then generates an alert describing the outlier and the correlations. The alert appears in the Problem Management Dashboard and may also trigger any other action possible for an alert. (See Actions.) The default action for an adaptive alert is to generate an SNMP trap on crossing.
As an example, the adaptive alert named 'Increased APPLICATION RESPONSE TIME (SERVERS) due to Overloaded Server(s)' is triggered when a Business Group containing servers has a simultaneous increase (outlier) in Application Response Time (Servers) and Connection Requests (Servers). In this event, Outbound Application Response Time and Inbound Connection Requests are said to be correlated.
In contrast, the adaptive alert named 'Increased APPLICATION RESPONSE TIME (SERVERS) due to Potential Server or Backend Problem' is triggered when an Application Response Time (Servers) outlier is identified, but there is no corresponding increase in Connection Requests (Servers), i.e., there is no correlation. This scenario would indicate a server that is taking longer to process requests for some reason other than increased network requests.

A SYN Flood DoS (Denial of Service) ATTACK is either; a) an attempt by a hacker to make an application inaccessible for legitimate client requests by overloading the application with an inordinate number of bogus connection requests or, b) a client application gone haywire. With legitimate connection requests there is typically some data transferred back to the client as a result of the completed TCP session. Adaptive Alert Analysis
The appliance automatically establishes an adaptive threshold for each metric involved in an alert analyses. This threshold is based on the moving average of the metric value over the latest time interval, known as the time window. The size of the time window can be adjusted using a parameter called “window size.” By reducing the window size for a specific adaptive alert, the moving average becomes more responsive to fluctuations in metric value, resulting in less alerts. All calculations performed in adaptive alert analysis are done using the 1 minute metric values.
The appliance uses an algorithm known as the outlier method to calculate the adaptive threshold based on the moving average for a metric. Two different outlier methods can be used to calculate the threshold:
Offset outlier method
The offset outlier method determines the adaptive threshold by adding an offset to the moving average. The size of the offset is determined by two parameters: sigma and threshold factor. Sigma is the standard deviation from the moving average over the time interval specified in the window size. The threshold factor is the number of sigmas added to the moving average to determine the adaptive threshold. The threshold factor can be modified for each adaptive alert using the alert manager.
Figure 13
Adaptive Alert - offset outlier method
adaptive threshold = µ + mó
where µ is the moving average, m is the threshold factor, and ó is the sigma
Multiplicative outlier method
The multiplicative outlier method determines the adaptive threshold by multiplying the moving average by the threshold factor. The threshold factor can be modified for each adaptive alert using the alert manager.
Figure 14
Adaptive Alert - multiplicative outlier method
adaptive threshold = mµ
where µ is the moving average and m is the threshold factor
The outlier method used depends on the variability of the metrics involved. Outliers for metrics that normally have low variability, such as Application Response Time for a typical HTTP server, are more readily detected using the multiplicative outlier method, rather than the offset outlier method, since the sigma (standard deviation) is normally quite low. This may not be true of Application Response Time for other servers, such as streaming media servers. When modifying adaptive alert analysis parameters, use a time series chart to examine the variability of the metric for the desired group and choose an appropriate outlier method. Running the analysis manually (as described in Running the Analysis Manually) can facilitate this process.
The maxOutlierValue parameter specifies the maximum value for the adaptive threshold, regardless of the outlier method used to determine the threshold. For instance, for groups with relatively high packet loss, the outlier method may calculate an adaptive threshold approaching 100%. In these instances, it may be desirable to specify that the adaptive threshold should never exceed 30% (or any another value of the user’s choosing) because this condition.
Naturally, changing the outlier method or the parameters that are used in determining the adaptive threshold affects the number of adaptive alerts that are triggered. A higher adaptive threshold results in less alerts, while lower thresholds results in more alerts.
Once the adaptive threshold is determined for a metric, the appliance can examine metric values to identify outliers. Outliers are metric values that exceed the adaptive threshold; however, not all outliers qualify as adaptive alerts. There are a number of analysis parameters (minimum duration, minCount, minSignifCount and minMeasurementValue) that affect this determination.
In some cases, an outlier should not trigger an alert unless the metric value exceeds the adaptive threshold for a minimum period of time. This can be configured using an analysis parameter known as minimum duration. The minimum duration, specified in seconds, identifies the minimum amount of time that the metric value must exceed the adaptive threshold before an adaptive alert is triggered.
To ensure that adaptive alerts are not biased by fluctuations observed in time intervals that have very little traffic, the analysis only considers time intervals that have a sufficient number of data measurements or packets. This behavior is controlled by the parameters minCount and minSignifCount respectively. The minCount analysis parameter specifies the minimum number of measurements required for an outlier to trigger an adaptive alert. This parameter can be modified for each applicable metric in the alert analysis. For instance, if minCount is set to 5 for Application Response Time, there must be 5 Application Response Time measurements in a 1 minute time interval for the alert analysis to include this data in the calculations. Anything less than 5 measurements is considered insufficient to be used in the analysis. Similarly, the minSignifCount parameter specifies the minimum number of packets that must be measured in a given 1 minute time interval for the alert analysis to include this data in the calculations.
The minMeasurementValue parameter specifies the minimum metric value that results in an adaptive alert. For instance, this parameter can be used to ensure that adaptive alerts based on packet loss are not triggered if the packet loss metric is less than 1% (or any other value of the user’s choosing).
Customizing Adaptive Alerts
The alert manager can be used to customize the parameters used in adaptive alert analysis. This section assumes a basic understanding of the alert manager. For more information, see Alert Manager.
The built-in adaptive alerts are listed in the alert manager as tasks of type “adaptive”. The alert manager can be used to modify the adaptive alert identity and analysis parameters. The group, schedule and metrics used in the alert analysis cannot be modified.
At present, new adaptive alerts cannot be created and the group assigned to each alert cannot be changed. However, the analysis parameters for existing alerts can be modified.

Modifying Adaptive Alert Analysis Parameters
The Condition tab of the alert manager is used to change the analysis parameters that trigger adaptive alerts (as described in Adaptive Alert Analysis). The condition tab provides two modes for modifying analysis parameters.
Basic Parameter Modification
The basic mode (default) can only be used to modify the window size and threshold factor. Advanced mode must be used to change other alert parameters.
Figure 16
Alert Manager - Condition tab - Basic Analysis Parameter Modification
 
In basic mode, the analysis window size can be changed using the slidebar near the top of the condition tab. The values displayed in the slidebar are normalized between 1 and 100. A larger window size results in more alerts since the moving average is less sensitive to fluctuations in the metric value. Advanced mode must be used to specify the window size in units of time.
The analysis threshold factor can be changed using the slidebar near the bottom of the condition tab. Each metric involved in the alert can have a different threshold factor. Select the desired metric in the drop-down list above the slidebar. The values displayed in the slidebar are normalized between 1 and 100. A larger threshold factor results in less alerts since the adaptive threshold is greater. Advanced mode must be used to specify the numeric value of the threshold factor.
Advanced Parameter Modification
Basic mode provides a simplified, conceptual means to change the sensitivity of adaptive alerts. Advanced mode provides a more flexible way to fine tune the behavior of adaptive alert analysis. Advanced mode can be used to specify the numeric value of analysis parameters. If necessary, review the explanation of the analysis parameters in Adaptive Alert Analysis.
To access advanced mode, click the Advanced checkbox in the Condition tab.
Figure 17
Alert Manager - Condition tab - Advanced Analysis Parameter Modification
 
The Metric field allows you to select each of the metrics used in the analysis for modification.
The following analysis parameters are applicable to all metrics.
Window Size—The window size is specified in seconds but is rounded to the nearest minute value. The window size cannot be less than 30 minutes and cannot exceed 12 hours. The default value is 7200 seconds (2 hours).
Minimum Duration—The minimum duration is specified in seconds.
The remaining analysis parameters can have different values for each metric involved in the adaptive alert analysis. Select the desired metric from the pull-down box. The metric selected determines which analysis parameters are available, since not all parameters apply to each metric.

outlierMethod—Set the value of outlierMethod to 2 to use the offset outlier method. Set the value of outlierMethod to 4 to use the multiplicative outlier method.
minCount—Set the minimum number of metric measurements required for a 1 minute time interval to be included in the analysis.
minSignifCount—Set the minimum number of packets required for a 1 minute time interval to be included in the analysis.
minMeasurementValue—Set the minimum value for a metric to be included in the analysis.
maxOutlierValue—Set the maximum value the adaptive threshold can be assigned by the outlier method.

Running the Analysis Manually
After modifying the analysis parameters, you want to see how your changes would have affected the number of adaptive alerts triggered. This allows you to run “what-if” analysis of parameter changes. These questions can be answered by running the adaptive alert analysis manually. To do so, click the Run Analysis button in the Condition tab.
The Desktop Console asks you which group to run the analysis against. Select a group of interest. The console then displays a Detailed Adaptive Alert View of the adaptive alert for the specific group using the new analysis parameters. This view shows the moving average, adaptive threshold and all outliers identified using the new analysis parameters run against metrics collected in the time selection of the project.
