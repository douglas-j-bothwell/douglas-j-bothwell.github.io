AppInternals Integration
This section describes creating integration links from the Desktop Console in AppResponse to the Desktop Console in SteelCentral AppInternals. As shown in the following figure, these integration links let users right-click on a system name or IP address and open the AppInternals Desktop Console. This Console displays detailed performance data related to that system for the same time period as in the AppResponse Desktop Console.
Figure 1
Viewing AppInternals Metrics from the AppResponse Console
 
The integration links use a script deployed on an AppResponse appliance as an Insight CLI extension. You download and install the Panorama Integration insight to create the script and a set of predefined integration (see Configuration and Setup for details). Once you install the links, you can adapt them for your purposes or create your own custom links (see Creating Custom Desktop Console Integration Links).
Note: Integration links to AppInternals requires AppResponse Version 8.0.8 or higher and AppInternals Version 7.1 or higher

How Links Request AppInternals Data
When a user follows an integration link, the link opens the AppInternals Desktop Console to display the data it requests. The AppInternals console must always connect to an AppInternals data source. Each data source (also called an “SMP”) monitors many systems (each running the AppInternals “DSA”).
What is the Active SMP?
For the data displayed in the AppInternals console to be relevant, the SMP that the AppInternals console uses should be monitoring the same system for which the user clicked the integration link.
However, integration links do not specify the SMP for the AppInternals console to use. It is important to understand the behavior that determines which AppInternals SMP the AppInternals console will connect to (the “active SMP”). The links follow these rules:
If the AppInternals console is already running, and it has projects open, it uses the SMP for the active project (and displays data in that project).
If the AppInternals console is already running, but has no project open, it connects to the SMP used by the last project opened.
If the AppInternals console is not already running, the link downloads it from the SMP specified in the AppResponse appliance interface (see Specify SMP to Use if AppInternals Console Isn’t Running), and connects to that SMP as the data source.
Recommendations
Because of this behavior, keep in mind these suggestions for effective integration links:
The IP host the user clicks on should have the AppInternals DSA installed.
The DSA must be managed by the SMP that the AppInternals console is connected to.
That DSA must have metric data available for
–
The requested time window (this is usually the project time specified in AppResponse when a user right-clicks on an IP).
–
The object pattern specified in the link.
If you have multiple SMPs, choose one SMP and configure it to manage all DSA hosts that support integration links to the AppInternals desktop console.
Alternatively, create one or more business groups in AppResponse that contain all DSA hosts in your observed network. Give them descriptive names such as “.NET hosts with links to AppInternals” or “Java hosts with DSAs.” (The primary purpose for this type of business groups is to provide a list of IPs that support integration links; the metrics collected for these groups might not be useful.)
In AppInternals, check each DSA and verify that it is collecting the data of interest.
Configuration and Setup
This section describes the configuration required to enable links from the AppResponse Desktop Console to the AppInternals console. This configuration requires editing a file on the system running the AppInternals console and steps in the AppResponse web UI and Desktop Console.
Configure Firewalls to Allow Access to AppResponse Appliance and AppInternals SMP
If a firewall separates the AppResponse appliance and the client host, verify that the appliance can communicate with the client over port 27402 (i.e., that the firewall is not blocking this port).
Similarly, if a firewall separates the client host and the AppInternals SMP, verify that the client can communicate with the SMP over port 3433.
Client System: Edit panorama.ini to Enable AppInternals Console Connections
Integration links in the AppResponse Desktop Console use a socket interface to connect to the AppInternals console running on the same system. You must enable this socket interface by editing the panorama.ini file
Edit the AppInternals console initialization file, panorama.ini, in the user application data directory Documents and Settings\<user>\Application Data\OPNET\Panorama (%APPDATA%\OPNET\Panorama). Follow these steps:
1)
If the AppInternals console is already running on the system, stop it. (The AppInternals console overwrites panorama.ini when it stops.)
2)
Edit the panorama.ini file. Search for the ipc element:
<ipc enable="false" port="27402" protocol="http" restrictToLocalhost="true"/>
3)
Change the value of the enable attribute from false to true, and the value of the restrictToLocalhost attribute from true to false:
<ipc enable="true" port="27402" protocol="http" restrictToLocalhost="false"/>
4)
Save the changes.
If you do not change the enable attribute (or if the AppInternals console is not already running), requests to the AppInternals console will fail with an Unable to Connect or similar message.
AppResponse Administration > System Web Interface: Specify DNS Server and AppInternals SMP
Specify these settings from the AppResponse appliance web interface. Log on to the web interface at https://<appresponse_xpert_system>:8443.
Specify the DNS Server
As described in Mapping IP Addresses in ACE Live to AppInternals DSA Host Names, some forms of integration links specify a question-mark placeholder. AppResponse substitutes the IP address that a user clicks and maps it to a system name that is presumably being monitored by AppInternals. For this mapping to work, you specify a DNS server to use.
In the Administration > System > Setup page, verify that the Domain Name Servers setting specifies at least one server. The integration links use the server to map IP addresses to system names.
Specify SMP to Use if AppInternals Console Isn’t Running
When users click an integration link in the AppResponse Desktop Console, it first checks if there is an instance of the AppInternals console already running on the local system. If there is not, the AppResponse Desktop Console connects to an SMP system to download and start a new instance of the AppInternals console. The AppInternals console then connects to the SMP to open a project.
You specify the SMP to use for downloading and connecting the AppInternals console in the Administration > System > Integration > AppInternals web page. In the Panorama SMP Hostname[:port] setting, enter the name or IP address of the system running the AppInternals SMP.
AppResponse Desktop Console: Install Link Template, Create and Test Default Links
The integration links use a script deployed on an AppResponse appliance as an Insight CLI extension. You use the AppResponse Desktop Console to install the insight. Unlike typical insights, the Panorama Integration insight only deploys the CLI extension script and creates an integration template on the AppResponse appliance.
Install the Insight
1)
Open the AppResponse Desktop Console and connect to the appliance. Choose Insights > Update Center. In the Update Center window, in the Server list, select support.opnet.com/ace_live/insights/panorama. (If that server address is not available in the Server list, click Add Update Server in the list and supply the address in Connect to Update Server dialog box.)
2)
Select the Panorama Integration insight and click Install. The insight disappears from the list. Click Close.
Create Integration Links
Once the template is installed, use the Integration Manager to actually create the integration links. The links are available on the client system where the AppResponse Desktop Console is running, regardless of which appliance the console connects to.
1)
In the AppResponse Desktop Console, choose Tools > Integration Manager.
2)
In the upper panel of the Integration Manager, right-click on the Panorama template you just installed (it appears with a Link Type of Template) and choose Create a Link from Template. This will create a set of AppInternals integration links (with a Link Type of Link) and open them in the lower panel.
Figure 2
Creating Links from an Integration Template: Example
 
In the lower panel of the Integration Manager, the Name field specifies a category label (Panorama, by default) for the set of links you just created. When you right-click on an IP address in a table, the links you just created will be grouped under this name. The Label fields specify the names of the individual links. You can change these names to suit your purposes.
3)
Click OK or Apply to save the links on the client system.
Test Your Links
Verify that the default AppInternals console links work as intended:
1)
In the AppResponse Desktop Console, choose Tools > Search and search on an IP address for a host with a DSA that:
–
Has one more data collectors running, AND
–
Is managed by the currently active SMP (see What is the Active SMP?).
2)
In the search results list, click on the Favorites Table link for the IP you just searched on.
3)
In the Favorites Table, right-click on the IP and click one of the new Panorama:
If the AppInternals console is not already running, the link opens an empty browser window and connects to the SMP specified in the Administration > System > Integration web interface of AppResponse (see Specify SMP to Use if AppInternals Console Isn’t Running for more details). The SMP downloads the AppInternals console, starts it, and prompts for sign-in credentials.
If the AppInternals console is already running, the link opens a new view in the active project (or in a new project if no project was open).

Creating Custom Desktop Console Integration Links
This section describes how to create custom integration links to the AppInternals console. The easiest way to do this is to adapt the links created after you installed the template (see Table 1). You can modify the existing links or use them as models for new ones.
Choose Tools > Integration Manager in the AppResponse Desktop Console to open the Integration Manager. Click the Panorama row with a Link Type of Link in the upper panel to see the details in the lower panel. For details on using the Integration Manager, click the help icon.
While there are additional options, you typically customize existing AppInternals console integration links by changing:
The view type to create in the AppInternals console (object, metrics, or deviation)
The object or metric patterns that specify the data you want the AppInternals console to request
Whether the pattern should include the specific host name that the user clicked on
Details of an Integration Link
Consider this integration link (this is the Server CPU Performance link from Table 1):
extension:launchPanorama -action createView -viewtype objects -start $start -end $end -ip $ip -pattern "Host(?) > * > CPU(_Total) %3B Host(?) > * > CPU(-Aggregate-)"
Here is a description of the clauses in this link:
extension:launchPanorama
All integration links contain this clause. It specifies the CLI script to invoke.
-action createView -viewtype objects
These clauses specify that the link will create a new AppInternals console view and the type of view to create. See the AppInternals console documentation for more details on the different views. Valid values for the -viewtype argument are:
deviations opens a Deviating Metrics view that displays metrics with values that have deviated the most from their expected range for the time range.
objects opens an Object and Metrics view that displays a collection of objects and values of one or more metrics associated with those objects.
metrics opens a Metrics view that displays a collection of metrics for one or more domain objects.
-start $start -end $end
These clauses specify that the AppInternals console will use the same time period as the AppResponse Desktop Console. The console replaces the $start and $end built-in parameters with the start and end time for the AppResponse project. Typically, all integration links use the built-in parameters as shown here. However, you can instead supply explicit time values specified as the number of seconds since 1970 (“UNIX time”). This is rarely useful.
-ip $ip
This clause is required only if the -pattern clause includes the question mark placeholder token. See Mapping IP Addresses in ACE Live to AppInternals DSA Host Names for details.
-pattern “Host(?) > * > CPU(_Total) %3B Host(?) > * > CPU(-Aggregate-)"
This clause specifies one or more object or metric patterns that will be passed to the AppInternals console. Note the following:
In this example, the question mark is a placeholder that the script replaces with the host name resolved from the -ip $ip clause. See Mapping IP Addresses in ACE Live to AppInternals DSA Host Names for details.
In this example, the %3B character string is a URL-encoded semi-colon that separates two object patterns.
See the AppInternals console online documentation for details on patterns. In summary:
–
An object pattern is a wildcarded string that can match multiple object instances. For example:
-pattern “Host(*) > Windows > CPU(*)”
–
A metric pattern is an object pattern that includes a metric type at the end, separated by a colon (:):
-pattern “Host(*) > Windows > CPU(*):Interrupts/Sec.”
–
Specify a saved filter name by preceding it with an “at” sign (@):
-pattern “@Systems”
If you omit the -pattern clause, the default is *, which requests data for all available object instances.
NOTE: URL Encode Special Characters in Patterns
You must URL encode special characters, such as percent signs (%) and semi-colons (;), in object and metric patterns. For example, to match the metric % Usage in a pattern:
-pattern “Host > Windows > CPU:%25 Usage”
Mapping IP Addresses in ACE Live to AppInternals DSA Host Names
You can configure integration links to translate an IP address that a user clicks to the host name for that IP address. In the object or metric patterns for the link, you specify a question-mark (?) placeholder in the pattern where you want to substitute the translated host name. The link replaces any occurrence of ? in the pattern with the host name.
This is useful when the IP address corresponds to a DSA system being monitored by the active SMP (see What is the Active SMP? for details). Supply a question mark for the Host instance name in the object pattern for the link, and the link will show users performance data for the system they clicked.
NOTE: Be Sure the System is Monitored by the SMP
If the IP address does not correspond to a DSA system being monitored by the active SMP, the data requested by the link will not match any data available to the AppInternals console. As a result, the AppInternals console will show the Your pattern returned no data error screen.
To map from the IP to the DSA host, a link must include the following:
1)
The -ip $ip clause. The AppInternals console replaces the $ip built-in parameter with the address the user clicks on, and uses the DNS server specified in the AppResponse appliance (see Specify the DNS Server for details) to translate it to a host name. (If the user clicks a host name, that name is used directly.)
2)
The -pattern clause with the string “Host(?)“. The script replaces the question mark with the translated host name and passes it to AppInternals.
The following example shows the required clauses in bold:
extension:launchPanorama -action createView -start $start -end $end -ip $ip -viewtype objects -pattern "Host(?) > * > CPU(_Total)"
To continue the example, suppose the network includes a host with IP address 192.168.12.132 and domain name workstation322. This host has a DSA running and is monitored by SteelCentral AppResponse. An end user opens an IP table in the SteelCentral AppResponse Desktop Console, right-clicks on this IP, and invokes the link. The following then occurs:
1)
ACE Live performs a DNS lookup on the IP 192.168.12.132.
2)
The link replaces the ? placeholder as follows:
–
If the lookup is successful, SteelCentral AppResponse substitutes domain name into the pattern string. Thus, in this example, SteelCentral AppResponse sends the following:
"Host(workstation322) > * > CPU(_Total)"
–
If the lookup is not successful, ACE Live substitutes the asterisk wildcard (that matches zero or more characters) into the pattern instead:
"Host(*) > * > CPU(_Total)"
3)
The script uses the completed pattern string in the HTTP request it sends to the active SMP.
