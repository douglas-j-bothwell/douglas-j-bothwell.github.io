Information Metric Sources
The information metrics are special metrics that provide additional data on a group. Data displayed in the information metrics comes from a variety of sources including DNS and the WHOIS Database (which you can edit using the AS Information Manager).
The AS Name, AS Number and AS Description metrics provide additional information for autonomous system groups: ISP AS, Peer AS and Dest AS. These groups are identified by their autonomous system number within the console. To complement the AS number, the appliance extracts the description of the autonomous system from the WHOIS database and makes it available in the metric named AS Description.
The Group metric identifies a group displayed within a specific view (e.g. “192.168.1.1”, “Total Traffic > UDP”).
The Group Membership metric identifies all the Business Groups an IP address is a member of, or all the Business Group Containers a Business Group is a member of.
The Information metric provides additional information about the group. If the group is an IP address, the appliance queries DNS and display the hostname associated with that IP address in the Information metric. If the group is a Business Group, the Information metric shows the description of the group configured in the Business Group Manager.
Information metrics can be added and removed from tables, but cannot be displayed in charts.

WHOIS Database
The WHOIS database is maintained by the routing information registries (RIRs), and contains information related to Autonomous Systems (AS). The appliance uses data from the WHOIS database to map IP addresses to AS numbers, and to map AS numbers to AS names. Since this information is displayed frequently in the console, the appliance uses a local static snapshot of the WHOIS database. The snapshot is constructed by merging data from the WHOIS databases from the RIRs.
This database is not applicable to AppResponse Standard.
The WHOIS directory is a distributed database that identifies Internet domains and IP addresses. The appliance uses data from the WHOIS directory to map IP addresses to AS numbers and to map AS numbers to AS names. Since this information is displayed frequently, the appliance maintains a local copy of WHOIS data rather than issue live queries to WHOIS servers. The local WHOIS database merges data available on multiple WHOIS servers maintain by Registrars.
Live queries to WHOIS servers can be issued using the WHOIS lookup manager (see WHOIS Lookup Manager). The results of live queries are displayed in the whois lookup manager, but do not update the WHOIS database stored on the appliance.
AS Information Manager
The WHOIS database on an appliance is updated with each software release. The database can also be changed manually using the AS Information Manager. To edit AS information in the Desktop Console, choose Tools > AS Information Manager.

The AS Information Manager can be used to configure the following information for each autonomous system.
AS Name—The name of the autonomous system identified by AS Number. The AS Name is displayed in the Information and AS Name metrics.
AS Description—The description of the autonomous system identified by AS Number. This information is displayed in the AS Description metric.
The WHOIS database on an appliance includes entries for all 65536 autonomous systems. As a result, the AS Information Manager does not provide a New button since AS numbers beyond 65535 are not allowed.
For performance reasons, the AS Information does not display all 65536 autonomous systems. Use the AS Range selector to choose the range of autonomous systems you wish to examine.
The appliance supports custom mappings from private IP addresses to AS numbers. For more information, see Setting Up Private-Address to AS-Number Maps.
TCP Metric Sources
The TCP metrics represent data captured in real time from TCP headers. The appliance understands the semantics of TCP and extracts meaningful information from TCP connections as they occur, directly from the network.

TCP Connection Metric Sources
TCP connection metrics are relative to communicating IP addresses identified as either TCP clients or TCP servers, without reference to the position of the appliance. The IP address that sends the first packet (the SYN packet) in the three-way handshake used to establish a TCP connection is assigned the role of TCP Client for that connection.
Thus, TCP connection metrics labeled TCP Client represent data from clients that initiate TCP sessions within the group. TCP Server metrics represent data from TCP servers within the group that respond to client connection requests. For instance, Connection Requests (TCP Server) represents the number of connection requests sent to TCP servers within the group. (Another way to define a TCP Server is that it is the machine listening on a well-known port for connection requests.)

TCP Turn Sources
Many applications establish a persistent TCP connection that subsumes many individual request-response interactions, or “turns,” between client and server. AppResponse TCP turn metrics use a generic, Layer-4-based model to deliver TCP turn metrics for such persistent connections that usefully approximate the user experience and other aspects of application performance.
A turn is a single request-response interaction. Recall that in TCP, connection-management and flow control are handled by packets (such as SYN, ACK, and FIN) that have a special flag set in the header. Most of these packets have a zero payload length, some do not. The latter, most usually an ACK packet, are generally carrying data representing a client request—this distinction is key to the AppResponse turns-based model. Each packet in the data sent by the server in response to a request has a data flag on it and a non-zero payload length; it is a data packet. The important thing to keep in mind here is that an ACK packet with non-zero payload length (an ACK that is also a data packet) is likely a client request.
In the AppResponse turns-based model, a reversal in the direction of data packets represents one-half turn (a request from the client and then the data sent by the server in response); the second reversal signals the beginning of a new turn by indicating a new request. (Pure ACKs—those with zero payload length—do not count for turn recognition.) The following figure shows this model.

Utilization Metric Sources
The utilization metrics represent data captured from the IP headers of any IP traffic on the network. These metrics are described in Utilization Metric Definitions.
Inbound/Outbound Utilization Metrics
All utilization metrics are recorded in each direction separately. Inbound and Outbound metrics represent the direction of traffic flow relative to a group. For example, Throughput (Inbound) provides a measure of the rate of traffic sent to the group, while Throughput (Outbound) provides the rate of traffic being from the group.
This rule applies to all groups with the following exceptions.
Utilization Direction based on the Internal Address List

For more information, see Internal Address List.
Utilization Direction based on BGP Information
For the following groups, Inbound/Outbound Utilization is based on the Internal Addresses list specified on the appliance (Administration > System > Setup > Internal Addresses):
ISP AS—An ISP AS group represents all IP addresses routed over a specific ISP (see Built-In Groups). Since the BGP information available to an appliance does not include inbound routes, inbound metrics for an ISP AS group are estimated based on the assumption that inbound traffic is using the same BGP path as outbound traffic. This is a fairly good estimate except for significantly multi-homed sites with specialized routing policies.
Peer AS—A Peer AS group represents all IP addresses routed over a specific ISP AS and then through a specific Peer AS (see Built-In Groups). BGP information available to an appliance does not include inbound routes. To estimate inbound metrics for a Peer AS group, AppResponse assumes that inbound and outbound traffic use the same BGP path. Because Peer ASes are by definition significantly multi-homed, inbound metrics for Peer AS groups are best treated as relative to other Peer AS groups.
When viewing Traffic and Throughput metrics, you can also view the sum of inbound and outbound data in a single metric.
