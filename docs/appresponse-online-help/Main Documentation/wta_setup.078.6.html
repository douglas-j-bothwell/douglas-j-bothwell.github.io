SSL Decoding: Enabling and Configuring
This section describes how to set up SSL decoding for HTTPS traffic for which you want to collect detailed page and/or user information. The following topics are discussed:
System Requirements
Workflow Description

System Requirements
Web Transaction Analysis supports SSL decoding for web servers that use the RSA encryption algorithm only. AppResponse cannot decode traffic from servers that use the Diffie-Helman key exchange algorithm.
If you do not know the algorithm used by the web servers in your network, contact your web master or IS organization.
Workflow Description
The following steps outline the general workflow for setting up SSL decryption for a specific set of web pages:
1)
If the encrypted web traffic includes any sensitive parameter values that you want to mask out, specify the parameter keys as described in Masking Private Data in Observed Web Traffic.
2)
If SSL decoding is not enabled on the appliance, enable it as described in Enabling SSL Decoding.
3)
Specify the private keys used by the SSL servers of interest, as described in Specifying Private Keys.
4)
It is good practice to check the web UI periodically to see if any HTTPS traffic of interest is not getting decrypted, as described in Periodic Updating (Recommended).
Enabling SSL Decoding
To enable SSL decoding, perform the following steps.
Procedure 3
Enabling SSL Decoding
1.
Configure AppResponse to mask out all sensitive information contained in key-value pairs that might appear in decoded packets (URLs, cookies, and POST messages):
1.1.
In the Desktop Console, choose Tools > Pages > Information Privacy Manager
1.2.
Specify all parameters that might contain sensitive information in the observed web traffic.
For more information, see Masking Private Data in Observed Web Traffic.
WARNING—There is no way to mask sensitive data after it is saved on an appliance. For this reason, you must specify all parameters before you enable SSL decoding.
2.
Go to the “Page Analysis Configuration” Page (Administration > System > Pages).
3.
Select the Enable SSL Decoding check box.
4.
At the bottom of the page, click the Apply button to apply your changes.
AppResponse starts decoding HTTPS traffic for the servers and ports specified under Enable Page Analysis.
If the Enable Page Analysis option is selected, AppResponse does the following:
• 
Collects page data for decrypted pages
• 
Collects user data for Page Families that are included in the Tracking Web Page Users: the User Session Tracking Manager.
Note: If AppResponse cannot decrypt some traffic as expected, it might be due to a missing certificate. For more information, see Periodic Updating (Recommended).
End of Procedure 3
 
 
Specifying Private Keys
The following procedure describes how to set up SSL decoding.
Procedure 4
SSL Decoding: Initial Setup
1.
Go to the “Page Analysis Configuration” Page (Administration > System > Pages).
The Private Key Configuration table displays a list of user-defined labels, which describe the set of private keys that AppResponse uses to decode observed SSL traffic. For AppResponse to decode SSL traffic from a specific server, you must add the private key for that server.
Note: After you add a private key to this list, no user can view or edit the content of that key (only the label and comment are visible). If you need to update a key for any reason, you must delete that key from the list and then add it again.
Figure 3
 Private Key List (Administration > System > Page)
 
2.
Select the Enable SSL Decoding option if it is not already selected.
3.
For each SSL server of interest, do the following:
3.1.
Obtain a copy of the private key for the server, if you do not have it already.
NOTE—You must have a copy of the private key in a directory that is accessible to the local host. The key must be in Open SSL PEM (Privacy Enhanced Mail) format.
3.2.
Click the Add button.
The Private Key options appear.
Figure 4
 Private Key Options (Administration > System > Page)
 
3.3.
Load a copy of the private key (the one used by the SSL server) onto the appliance:
• 
If necessary, save a copy of the private certificate in a location that is visible to your local host.
• 
Select either the Copy & Paste or the Upload radio button.
• 
Copy (or upload) the certificate copy.
3.4.
Specify the following options:
• 
Label—A descriptive label for the key.
• 
Comment—The label and comment are the only information that will appear for the key after you create the entry. Include any additional “public” information you would like to appear for this key in the Comment field.
• 
Password—The password for the private key, if the key is password-protected. (This string is masked when it is entered by the user and is not saved on the appliance.)
3.5.
Review your settings and then click OK (bottom-right corner).
If the private key is encrypted, AppResponse uses the password you entered in the previous step to decode the key.
AppResponse re-encrypts and stores the private key on the AppResponse appliance. This encryption is internal to AppResponse to ensure that the key is inaccessible to any user.
The Private Key list is updated with an entry for this key.
Note: After you add a private key to this list, no user can view or edit the content of that key (only the label and comment are visible). If you need to update a key for any reason, you must delete that key from the list and then add it again.
4.
To apply your changes, click the Apply button at the bottom of the page.
End of Procedure 4
 
 
Periodic Updating (Recommended)
The SSL Servers with Missing Private Keys list (Administration > System > Pages) shows all servers that have sent web-over-SSL traffic that AppResponse has observed, but could not decode because the private key is not included in the  Private Key List (Administration > System > Page).
Figure 5
“SSL Servers with Missing Private Keys” List
(Administration > System > Pages)
 
Note: It is good practice to check this list periodically for servers whose HTTPS traffic is not getting decrypted.
If you want AppResponse to decrypt traffic from a listed server, do the following:
1)
Add the private key for that server to the Private Key Configuration list.
2)
Select the server in this list and click Delete (below).
3)
Click Update (at the bottom of the page) to confirm this change.
The server will not re-appear in this list unless the private key is incorrect or gets changed in the future.
If you do not want AppResponse to decrypt traffic from a listed server, do the following:
1)
Select the server in this list and click Ignore.
2)
Click Update (at the bottom of the page) to confirm this change.
This server gets added to the Ignored Servers list.
