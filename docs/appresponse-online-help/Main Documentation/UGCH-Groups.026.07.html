VLANs
VLAN is short for virtual LAN, a network that acts as if all the computers are connected to the same physical LAN even though they may actually be physically located on different segments of a physical LAN. The main purpose is to subdivide a LAN to provide traffic privacy, or quality, or priority, separate from the rest of the LAN traffic. For example: e-mail traffic goes to corporate, web surfing traffic goes out the nearest POP, and voice traffic goes to your voice gateway.
VLANs are configured through software rather than hardware, which makes them extremely flexible. A big advantage of VLANs is that if a computer is physically moved to another LAN, it can stay on the same VLAN without any hardware reconfiguration.

VLAN Grouping
VLAN Grouping an be useful when you want to view and collect metrics for VLAN-specific flows in the VLANs table. When ystem > Administration > System > Advanced Data Collection Options > VLAN Grouping< Enable VLAN Grouping is enabled, the appliance processes packets for each flow separately based on the VLAN ID. This results in multiple flows (one per observed VLAN) for each IP/port pair.
Note the following:
These VLAN-specific flows are visible in the VLAN table (View > Tables > VLANs) only; all other windows include only the flow with the highest observed bidirectional throughput.
Enabling this option can add significant workload to the appliance—potentially doubling, or more, the number of flows processed.
If this option is disabled (the default) and the Duplicate Packet Filter is enabled on the System > Setup page (also the default), the appliance ignores VLAN IDs and retains the first copy of each packet regardless of the VLAN ID.
If you want to use this option, the monitored traffic must include 802.1Q packet header information.
The System > Administration web UI does not allow some sets of options to be enabled at the same time. If the VLAN Grouping checkbox is unavailable (grayed out), check to see if any of the following are enabled:
–
System > Pages > Enable Page Analysis
–
System > Interface Groups > Collect data for Monitoring Interface Groups
NOTE: If Packets for the Same Connection are Observed in Different VLANs, Some Traffic Might Be Ignored
This section describes how VLAN Grouping can affect the metric results for individual connections. In this context, a connection is the set of packets exchanged between two hosts over the same TCP port.
If VLAN Grouping is enabled, you might see inaccurate metrics for a connection if
The connection transfers packets across multiple VLANs, and
The appliance does not observe all packets in any one VLAN—that is, packets for that connection are split across multiple VLANS.
Key Concept—With VLAN Grouping enabled, metrics for each connection are based only on packets in the VLAN where the most packets were observed. Packets observed in other VLANs are ignored.
When an appliance observes packets for the same connection in different VLANs, it calculates metrics for that connection as follows:
VLAN Grouping off:
The appliance
–
Discards duplicate packets (==> the same packet observed in multiple VLANs).
–
Calculates metrics based on all unique packets observed for each connection (regardless of the VLAN in which a particular packet was observed).
VLAN Grouping on:
The appliance
–
Considers the packets for the “highest-packet” VLAN (that is, the VLAN in which the most packets for an individual connection were observed).
–
Ignores all other packets for that connection that were observed in other VLANs.
–
Calculates metrics based only on the packets in the highest-packet VLAN.
This is not an issue if the appliance sees all packets for a connection in one VLAN or another. It can be an issue if some packets are observed only in a VLAN that is not the highest-packet VLAN.
The most common real-world example of when this can occur is when a bidirectional connection between two hosts uses separate unidirectional flows that span different VLANS. In Figure 21, the appliance observes 300 packets for a connection: 100 packets (A --> B) in one VLAN and 200 packets (A <-- B) in another.

If VLAN Grouping is turned off, the appliance considers the packets from both VLANs and collects metrics based on all 300 packets.

If VLAN Grouping is turned on, however, the appliance ignores the A-to-B and considers only the B-to-A packets. In our example, this means that metrics for this connection are based only on the 200 packets from B to A and therefore might be inaccurate.

VLAN Information Manager
The VLAN Information Manager is used to create or delete groups of existing VLANs that can then be monitored using the Desktop Console.
Note: For the appliance to collect and show VLAN data, VLAN Grouping must be enabled. This option is disabled by default.
Figure 24
VLAN Information Manager
 
Once the VLANs are entered into VLAN Information Manager, the data can be viewed on the VLAN table. The table displays all VLANs listed by VLAN ID. The VLAN Information Manager gives you the ability to assign names to VLAN IDs, such as VLAN-101 VoIP traffic, VLAN-50 Engineering, etc. If the you configure alternate names other than the VLAN ID, these names appear on the table instead of the IDs.
For more information see VLAN Drill-downs.
When you create a new VLAN profile on a director, the profile is distributed to all the appliances in the domain. If an appliance already has a VLAN name for the same VLAN number, it gets overwritten the director.
