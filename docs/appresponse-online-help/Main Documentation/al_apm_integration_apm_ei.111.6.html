SNMP System Monitor Insight
The SNMP System Monitor Insight is useful for monitoring the performance of specific systems. It works with either of two data sources:
Basic SNMP Collector (built into AppResponse)
NetSensor (add-on module)
Which data source is used depends on how AppResponse is configured in the Administration > System > Element Polling > Integration web page. For details, see Setup and Configuration for Element Polling Insights.
When using NetSensor as a data source, this Insight provides four types of information about the system being monitored. The information appears on these tabbed pages:
Description—This page displays general information about a system: name, uptime, description, and location.
For more information, see Viewing a System Description.
Interface Status—This page displays status information about the interfaces on a specific system: state changes and last-known state. You can also view time-based charts of administrative and operational status.
For more information, see Viewing Interface Status.
Interface Traffic—This page displays performance information about the interfaces on a specific system: names, IP addresses, utilization, throughput, packet drops, and errors. You can also view time-based charts that show information for a specific interface.
For more information, see Viewing Interface Traffic.
Process Monitor—This page displays information about the processes on a specific system: system CPU usage, top processes, and memory and CPU usage for selected processes.
For more information, see Viewing System Processes.
When using the Basic SNMP Collector, most of the same information is available. However, because AppResponse is polling the systems directly, the following restrictions apply:
Metrics that are calculated by NetSensor (such as utilization) are not available.
No information is available on the Interface Status page. (Interface status is shown on the Interface Traffic page, however.)
The available data is time-limited:
–
No data is available from before the system was initially polled.
–
Historical data is discarded if the system is not polled at least once an hour. (Polling must be triggered manually.)
–
Historical data is kept in a rolling buffer for a maximum of 4 hours.
For additional details about using the Basic SNMP Collector, see Considerations When Using the Basic SNMP Collector.
Updating Data in the SNMP System Monitor Insight
There are various ways of updating the displayed data, depending on what data source you are using.
When using NetSensor, there are two ways to update the displayed data. (These are the same methods used by other Element Polling Insights.)
Manual update—Click the Update Now button.
Automatic update—Open the Project Time dialog box and select the Automatic Update option.
When using the Basic SNMP Collector, only manual update is possible:
On the Add New System page—Click the Go button.
On the Polled System List page—Deselect, then reselect the system.
Selecting a System to Monitor
Depending on your data source, use one of the following procedures to select the system you want to monitor.
Procedure 3
Selecting a System (NetSensor Method)
1.

1.1.
Choose Insights > Element Polling > SNMP System Monitor.
1.2.
Choose the system to monitor from the Polled Systems List.
Note: If the system you are interested in does not appear in the list, you must add it to a polling profile in APM Element Interface. The Add New System tab is non-functional when using APM EI.
2.
Alternatively, you can open this Insight from any individual IP address in the Desktop Console:
2.1.
Right-click on the address.
2.2.
Choose Analyze In > Element Polling > SNMP System Monitor.
3.
View the desired information, as described in
• 
Viewing a System Description
• 
Viewing Interface Status
• 
Viewing Interface Traffic
• 
Viewing System Processes
End of Procedure 3
 
 
Procedure 4
Selecting a System (Basic SNMP Collector method)
1.
Choose Insights > Element Polling > SNMP System Monitor.
2.
If the system you want to monitor appears in the Polled Systems List, select it. Otherwise:
2.1.
Click the Add New System tab.
2.2.
Enter a system (name or IP address) and community string, then click Go.
Note—If the community string uses any characters other than alphanumerics, see Handling Special Characters in Community Strings for how to enter them.
AppResponse contacts the specified system and, if necessary, starts an SNMP polling process on that system. The system will appear in the Polled System List the next time you open this Insight (if polling is still active).
3.
View the desired information, as described in
• 
Viewing a System Description
• 
Viewing Interface Traffic
• 
Viewing System Processes
Note: If polling has just been started on the system, it might be several minutes before data becomes available for display.
End of Procedure 4
 
 
Handling Special Characters in Community Strings
The following restrictions apply to community strings used with the Basic SNMP Collector:
These characters must be preceded by a backslash when they appear in a community string:
( ) & | \ < > ; ` ' "
For example, a community string of pub&lic must be entered as pub\&lic.
These characters are not allowed in a community string:
$ /

Viewing Interface Traffic
Use this procedure to view performance information for an interface.
Procedure 7
Viewing Interface Performance for a Specific System
1.
If necessary, open the SNMP System Monitor Insight and select a system using one of the methods described in Selecting a System to Monitor.
2.
Click the Interface Traffic tab.
The Interface Traffic tabbed page appears. The Interfaces table shows the interfaces on the monitored system and data for each interface. The data shown depends on the data source:
• 
NetSensor: utilization in and out, throughput in and out, percentage of errors, and percentage of dropped packets.
• 
Basic SNMP Collector: throughput in and out, percentage of errors, percentage of dropped packets, and operational status.
Note: Utilization values are estimates based on the speed of the interface. The reliability of this calculation depends on the accuracy of the interface speed returned to NetSensor.
Note: When using the Basic SNMP Collector, it might take several minutes to obtain the data necessary to calculate the Drops and Errors metrics.
Figure 9
“SNMP System Monitor” Insight > “Interface Traffic” Tabbed Page
 
3.
At this point, you can do the following:
• 
Display the most recent interface data from the system. (See Updating Data in the SNMP System Monitor Insight for details.)
• 
View data over time for a specific interface:
Select an interface in the Interfaces table. (You cannot select multiple entries in this table.) The charts show data for this interface over the total available time window. Which charts are shown depends on the data source. The charts are (top to bottom):
• 
NetSensor: utilization in and out, throughput in and out, percentage of errors, and percentage of dropped packets.
• 
Basic SNMP Collector: throughput in and out, percentage of errors, percentage of dropped packets, and operational status.
End of Procedure 7
 
 
Viewing System Processes
Use this procedure to view system processes.
Procedure 8
Viewing System Processes on a Specific System
1.
If necessary, open the SNMP System Monitor Insight and select a system using one of the methods described in Selecting a System to Monitor.
2.
Click the Process Monitor tab.
The Process Monitor tabbed page appears. The Top Processes table is updated to show the system process data collected from the system.
Note: The System Idle Process ID appears as 1 in this table, even though this ID appears as 0 in the Windows Task Manager. All other process IDs are accurate.
The System CPU Use chart is updated to show the CPU usage data collected from the system.
Note: When using NetSensor, the process table is calculated only for the first 24 hours of the project time.
Note: When using the Basic SNMP Collector, it might require several minutes to obtain the data necessary to calculate the CPU and Memory metrics.
Figure 10
“SNMP System Monitor” Insight > “Process Monitor” Tabbed Page
 
3.
At this point, you can do the following:
• 
Display the most recent interface data from the system. (See Updating Data in the SNMP System Monitor Insight for details.)
• 
View CPU and memory use for a specific process:
Select one or more processes in the Top Processes table for display in the Memory and CPU Use charts. If you select multiple processes, their CPU and memory use values are aggregated into single lines in the charts. This is helpful in cases where several processes form a working group and you want to understand the impact of this working group on the server.
• 
View process information for a specific time window:
Select a time window by dragging horizontally in the System CPU Use chart (top right). The Top Processes table is updated to show process information for the selected time window only.
End of Procedure 8
 
 
Considerations When Using the Basic SNMP Collector
When used with the Basic SNMP Collector, the SNMP System Monitor Insight relies on data from SNMP polling processes on each system that is being monitored. Data is shown only when specifically requested by the user and requires at least two polling cycles before data becomes available. This means that, when you make the initial data request, you might experience a delay before the requested data is available. You might also see only partial data until the Insight is able to collect a full data sample.
When you connect to a system from this Insight for the first time (by entering it on the Add New System page), the Insight initiates an SNMP polling process on the system. This polling process continues running on the appliance even if you close the Insight. It stops automatically one hour after the last update (that is, one hour after the last time you click Go in the Insight window).
When using the Basic SNMP Collector, the Polled Systems List displays a list of systems that are currently running an SNMP polling process. When the polling process stops on a system, that system no longer appears in the list.
Polling
Some important points about polling with the Basic SNMP Collector:
Polling begins when data is first requested for a system.
After polling is started on a system, at least two polling cycles are needed before full data becomes available. Because of this, you might experience a delay or see only partial data until a full data sample has been collected.
You must update a polled system at least once an hour or the polling process will be stopped and historical data discarded. To update the system:
–
From the Add New System page: click Go.
–
From the Polled System List page: deselect, then reselect the system.
Time Window
The effective time window and project time behave differently when using the Basic SNMP Collector:
The maximum possible time window—that is, the time window for which process information is available—is determined by the length of time that the appliance has collected data from the system. This means that, the first time you start polling a system, you might need to wait a few minutes for the appliance to collect a significant data sample.
The time window is largely unrelated to the project time. The data will reflect the project time only if the project time is within the effective time window for which data is available.
You can determine the effective time window by looking at time axis values in any of the time-based charts.
Troubleshooting this Insight
Problem:
When using the Basic SNMP Collector, the SNMP System Monitor Insight does not appear to be retrieving data from a specific system.
Solution:
Do the following:
1)
If you specified a system on the Add System Name tabbed page, verify that the System and Community values are correct.
Note—If the community string uses any characters other than alphanumerics, see Handling Special Characters in Community Strings for how to enter them.
2)
If the values in step 1 were correct or if you selected a system from the Polled System List, SNMP might not be configured properly on the system of interest. In this case, do the following:
a)
If you have root access to the SteelCentral AppResponse appliance, log in to the appliance as root using an SSH-enabled utility such as putty.
If you do not have root access to the SteelCentral AppResponse appliance, log in to another host (such as a UNIX workstation). This host must have connectivity with the system of interest.
b)
Enter one or both of the following commands, depending on the type of data you want to retrieve:
Process data:
snmptable  -Ci  -v 2c -c  <community_string>  <system> .1.3.6.1.2.1.25.4.2
Interface data:
snmptable  -Ci  -v  2c  -c  <community_string>  <system> iftable
