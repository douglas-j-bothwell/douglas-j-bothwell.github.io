Tracking Web Page Users: the User Session Tracking Manager
A User Session Tracking profile defines a set of related web pages and the key strings that the web client and server use to track individual sessions. AppResponse requires this profile to track individual users and the page views for each user.
To define these profiles, choose Tools > Pages > User Session Tracking Manager in the Desktop Console. You must define one profile per web application of interest; in this context, a “web application of interest” is an application for which you want to track individual users.

Important Notes
Note the following:
AppResponse assumes that usernames and session IDs are embedded in the web traffic as key-value pairs separated by the ‘=’ character (for example, username=jsmith and sessionID=1131).
If a web application uses different conventions for transferring usernames and session ID strings, you must define the keys using regular expressions as described in Defining Keys Using Regular Expressions. To use this method, you must have a working knowledge of regular expressions and the data formats used by the web application of interest.
Do not use any of the following characters in a User Session Tracking profile, unless you are defining a regular expression as described in Defining Keys Using Regular Expressions:
–
.           (period)
–
+          (plus sign)
–
*          (asterisk)
–
?          (question mark)
–
( )      (parentheses)
–
{ }      (brackets)
–
[ ]     (square brackets)
–
^           (caret)
–
$          (dollar sign)
–
\           (backslash)
Collecting user data can be highly resource-intensive, especially on large networks with a lot of web traffic. Therefore, it is good practice to collect user data only for Page Families of interest.
Defining a User Session Tracking Profile
The following procedure describes how to create a User Session Tracking profile for a web application.
Procedure 5
Creating a User Session Tracking Profile for a Web Application
1.
Capture a login of the web application in a packet trace file:
1.1.
If the application login exchanges passwords or other sensitive information, verify that is configured to mask out this information.
For more information, see Masking Private Data in Observed Web Traffic.
1.2.
If the application login is SSL-encrypted, set up AppResponse decryption.
For more information, see Enable SSL Decoding.
1.3.
Using tcpdump or Transaction Analyzer, capture the login sequence as follows:
• 
Go to the login page for the web application.
• 
Start the capture.
• 
Enter the username/password and log in.
• 
Wait until the "login-succeeded" page downloads completely.
• 
If the web application uses separate servers for logins and for content, download a few more pages so that you can capture data for the different servers used by the application.
• 
Stop the capture.
2.
Open the resulting packet trace file in the Transaction Analyzer Transaction Analyzer or Wireshark.
3.
In the Desktop Console, choose Pages > User Session Tracking Manager.

Associated Pages
If the web application uses the same server (or set of servers) for logins and other types of transactions, you can use the same URLs and servers for Associated Pages as you did for Login Pages.
If the application uses separate login servers, you will need to use the same workflow that you used for the login pages:
• 
Capture one or more associated page views using tcpdump or Transaction Analyzer.
• 
Examine the POST message(s) for the TCP port and IP address for the web server.
• 
Examine the POST message(s) to the URL host and path.
When you specify the host and path, your goal is to enter a regular expression that matches all page views of interest for the web application of interest (and no other web applications that you are tracking in AppResponse). The following are some example hostnames/paths:
• 
inet.acme.com and */*calendar* (“Any page view whose URL includes the host string inet.acme.com and a path with the parameter string calendar”)
• 
calendar.acme.com and * ( “Any page view whose URL includes the host string calendar.acme.com”)
• 
www.webstore.com and /shopchart/* (“Any page view whose URL includes the host string www.webstore.com and a parameter string that starts with /shopcart/”)
Parameters > User Key
Note: This section describes how to find and enter keys that are encoded as key-value pairs separated by equals signs (username=jsmith). If the application uses a different format, you must enter the key as a regular expression. For more information, see Defining Keys Using Regular Expressions.
Using the Find Packets operation in Wireshark or Transaction Analyzer, search for packets that contain the username string you specified when logging in. Make sure that you search the text/string/ASCII data rather than the binary/HEX data.

Parameters > Session Key
Note: This section describes how to find and enter keys that are encoded as key-value pairs separated by equals signs (username=jsmith). If the application uses a different format, you must enter the key as a regular expression. For more information, see Defining Keys Using Regular Expressions.
Look in the cookie field of the first POST message for the “login submitted” page. The cookie might include several key-value pairs that are possible candidates for the “correct” session key. Here are some possible clues that a specific key is the one you're looking for:
• 
The key includes some form of "session" (examples: JSESSIONID, <string>_WebSession, <string>_sessid).
• 
The value is lengthy and includes at least some numeric characters.
These clues are valid for many, but not all, web applications.
Note: You might find that the cookie includes several keys that are possible candidates. In the following example, the cookie contains two different key strings, ACME_WebSession_R and ACME_WebSession:
HTTP: cookie: __utma=12570986.195726709.1192744122.1246891527.1288982137.9; __utmz=12570986.1288982137.9.1.utmcsr=(direct)|utmccn=(direct)|utmcmd=(none); ACME_WebSession_R=0751031; ACME_WebSession=07510310
In this case, you might need to test the different keys before you find the key that results in the desired user tracking behavior, as outlined in the following steps.
Note: A web application might use “dynamic” session keys that vary slightly with each session. In this case, you can use a wildcard (‘*’) character with a common string that all session keys share. For example, a web application might use session key/value pairs such as:
MedSupplySalesAppSessionID_afjib=<session_id_1>
MedSupplySalesAppSessionID_djekg=<session_id_2>
MedSupplySalesAppSessionID_iopru=<session_id_3>
MedSupplySalesAppSessionID_dbhfk=<session_id_4>
In this case, you could use the expression MedSupplySalesAppSessionID_* to track all sessions for this application. Make sure that the wildcard expression is not too broad, and that it matches only session IDs for the web application of interest. Thus the expression MedSupplySalesAppSessionID_* is a good match for the web application in our example, while *SessionID* would be too broad.

6.
Go to Administration > System > Pages and verify that the “Enable Page Analysis” option is selected.
7.
Verify that the User Session Tracking profile you specified works as intended:
7.1.
Go to the login page of the web application and log in (if you are already logged in, log out and then log back in).
7.2.
Download several pages in a “typical” session for that web application.
7.3.

7.4.
Open the Web Application table (View > Table > Web Application).
7.5.

You should see the pages for your web session, listed under the username you used to log in.
8.
If the User table does not show the page views for your web session, repeat step 4. through step 7. until you get the results you expect.

Defining Keys Using Regular Expressions
AppResponse can automatically find keys that are encoded in the application/x-www-form-urlencoded format: key=value pairs separated by ampersands—for example, key1=value1&key2=value2.
If a web application uses a different encoding format, you can enter regular expressions to find the username, session ID, and other keys. This requires the following:
A working knowledge of regular expressions
A knowledge of the exact key-encoding format used by the application you want to monitor. Information about non-urlencoded formats is outside the scope of AppResponse documentation.
You can specify regular expressions in the following fields only:
Desktop Console > Tools > Pages > Information Privacy Manager > Definition tab > Key field
Desktop Console > Tools > Pages > User Session Tracking Manager > Parameters tab > Name of User Key field
Desktop Console > Tools > Pages > User Session Tracking Manager > Parameters tab > Name of Session Key field
Required Format
To find a non-urlencoded key, you must enter a regular expression in the following format:
1)
[regex] - This tag tells the appliance that you are defining a regular expression rather than a key string.
2)
A regular expression that tells the appliance how to find the matching string
3)
A parenthesized expression that tells the appliance how to find the key within the matching string
For example, suppose a web app defines usernames in the format
username:jsmith,password:XXXXXXX
Figure 13
Regular Expression to Find a non-urlencoded Key: Example
 
In this case, you would do the following:
1)
Go to the Desktop Console > Tools > Pages > User Session Tracking Manager
2)
In the Name of User Key field, enter [regex]username:([^,]*)
This says: find the string username:[all-chars-until-the-next-comma]
The appliance finds the matching string: username:jsmith
The appliance uses the parenthesized expression ([^,]*) to find the username: jsmith
