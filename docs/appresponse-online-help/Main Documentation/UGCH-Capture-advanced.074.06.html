Alert Snapshots
An alert snapshot is a saved copy of the traffic for a specific alert. The Minimum Retention Time specifies a guaranteed time window for retaining snapshot traffic. Snapshots are especially useful for investigating intermittent problems that cannot be anticipated in advance or reproduced.
Note: You can create snapshots for Static-Any alerts only.

Packet Storage Buffers: Rolling Buffer and Snapshot Buffer
An appliance with Snapshot functionality divides the total disk space for packet storage into two separate buffers:
The Standard Rolling Buffer, which stores packets as they are seen and captured on the monitoring interfaces. When the entire rolling buffer is filled and new packets are captured, AppResponse overwrites the oldest packets.
The Snapshot Buffer, which provides long-term storage for packets associated with specific alerts. You can configure alerts to generate “snapshots” of associated packets, which are stored in this archive at the end of each alert episode.
Key Concept—The packet retention window is the time window for which you can view captured packets before they are overwritten. This window is bounded by the oldest captured packet in the buffer (start window) to the most recent packet (end window).

Because it stores only packets needed for long-term viewing and analysis, the Snapshot Buffer can extend the overall retention window by up to ten times—for example, from four days for the Rolling Buffer alone to nearly forty days for both the Rolling and Snapshot Buffers combined.

Robust Troubleshooting Using Snapshots
The Snapshot Buffer acts as a long-term archive for packets associated with specific alerts of interest. The general workflow is to create special alerts in the Alert Manager (Tools > Alert Manager in the Desktop Console) that generate snapshots of problems or behaviors that you want to study over time. This is especially useful for problems that cannot be reproduced or that occur intermittently or unpredictably. You can configure an alert to generate a snapshot each time the problem occurs, store multiple snapshots in the Snapshot Buffer, and then analyze the problem as it occurs over time. This approach is often more effective than analyzing single alerts in isolation shortly after they occur.
You can specify a Minimum Retention Time for snapshots in the Administration > System > Capture page. This setting specifies the number of hours that a snapshot is guaranteed to be available.
Note—You can generate snapshots for alerts of type Static-Any only. In addition, the primary group for a Static-Any alert can include IPs and Business Groups only. If the Alert Manager > Group Tab > Group menu (top) or Including field (top) specify anything other than IP Addresses or Business Groups, the Snapshot tab is unavailable.

Generating Alert Snapshots
The following procedure describes how to configure an alert so that it creates snapshots when that alert is triggered.
Procedure 2
Generating Snapshots for an Alert
1.
Open the Desktop Console and connect to an appliance.
2.
Open the Alert Manager (Tools > Alert Manager).
3.
Select or create an alert of type Static-Any. (This is the only type of alert that supports Snapshots.)
For specific information about static alerts, see the following section in the AppResponse User Guide:
Alerting and the Dashboard > Alert Manager > Static Alerts
4.
Click the Actions horizontal tab and then the Snapshot vertical tab.
Figure 6
Snapshot Tabbed Page in the Alert Manager
 
5.
Specify the desired options for the alert (see Table 2).
6.
Confirm your changes by clicking Apply.
End of Procedure 2

Specify any additional IP addresses whose traffic you want to include in the snapshot (that is, IP addresses in addition to the addresses that triggered the alert).
This table should include all additional IP addresses that correspond to tiers for the application transaction of interest. This set of addresses might be larger than the set of addresses that can actually trigger the alert. For example, you might specify an alert for a three-tiered application that is triggered by problematic traffic between client and front_end_server. However, you will also want to include traffic between front_end_server and back_end_server in the snapshot.
You can use the ‘*’ and ‘?’ wildcard characters to include multiple addresses, but you cannot specify ranges (<address_a> - <address_b>).

Guidelines for Maximizing Packet Retention
To maximize packet retention in the Rolling Buffer, we recommend the following:
Limit the number of bytes per packet that are captured.
The Packet Size Limit setting (Administration > System > Capture web page) specifies the number of bytes per packet. By default, this setting is set to Unlimited. You might want change this setting to extend the retention window. If you set the packet size limit to 500, for example, you could extend the retention window by a factor of 2 or 3.
NOTE—Lowering the Packet Size Limit can extend the packet retention window significantly, but there are important trade-offs. Before you change this setting, read the description of this option in Table 1.
To maximize packet retention in the Snapshot Buffer, we recommend the following:
Configure Static-Any alerts carefully so that they only include IP addresses of interest.
If you find that the packet retention window is shorter than you would like, this is a sign that you need to reconfigure your snapshot-enabled alerts. Specifically, you should ensure that
–
Each alert captures only packets that are directly related to the behavior or problem that the alert is intended to detect. (Not only does this extend the packet retention time window; it also makes the problem of interest easier to troubleshoot because there is less irrelevant traffic to filter out.)
–
If snapshots are getting generated frequently, you should examine the condition defined for that alert. The condition might be too “sensitive” so that the alert gets triggered even for borderline cases.
