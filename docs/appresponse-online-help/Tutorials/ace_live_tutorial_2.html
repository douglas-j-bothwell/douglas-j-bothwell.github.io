Trending, Alerts, Reporting, and SLAs
This tutorial shows you how to
Use the Audit Insights to gain a high-level view of the traffic patterns and applications in your network
Determine useful thresholds for acceptable performance in your network
Configure and generate alerts, reports, and SLAs in the Desktop Console
You can perform this tutorial on any AppResponse appliance or AppResponse Rover.
Note: All AppResponse data is based on live network traffic. The data that appears in your Desktop Console will be different from the data shown in these Tutorials. The screen shots and other traffic-specific results shown in this tutorial are intended as examples only.
Prerequisites
This tutorial assumes that you have completed “AppResponse Tutorial: Getting Started.”
This tutorial also assumes that you, or someone in your organization, has already done the following:
1)
Racked and powered up the AppResponse appliance
2)
Set up the appliance using the Command Line Interface and the Web User Interface
3)
Created a user account with administrative privileges on the appliance (you will need these privileges to perform some steps)
4)
Verified that the appliance is receiving traffic:
–
(AppResponse appliance) The span ports are configured and are feeding traffic into at least one of the monitoring interfaces
–
(AppResponse Rover) The Rover can see traffic on the virtual interfaces
5)
Defined a set of Business Groups that reflect the areas of interest in your network (as described in the first AppResponse “Getting Started” tutorial)
Viewing Overall Network Performance Using Audit Insights
Audit Insights enable you to view traffic on your network and to break down that traffic into the different applications, clients, and servers that are using the network. You can use Audit Insights to answer questions such as:
What applications are on my network?
How much traffic is each application generating?
What clients and servers are using those applications?
In this section, you will use two auditing Insights to gain a high-level view of the overall traffic patterns and applications in your network.
Viewing Overall Traffic Levels
One of the best Insights to get started with is the Total Traffic Dashboard. This Insight shows the total amount of traffic seen by the AppResponse appliance.
1)
Launch the Total Traffic Dashboard Insight from the AppResponse Index by clicking on it in the Index.
–> The Total Traffic Dashboard Insight appears.
Figure 1 “Total Traffic Dashboard” Insight > “Time Series” Tabbed Page (EXAMPLE)
 
2)
Confirm that you are seeing the amount of traffic you expect.
The Total Traffic Dashboard shows basic information about the traffic on your network, such as throughput, round trip times, and packet loss.
a)
Note the amount of traffic seen.
If this is zero or very low, check to see if your span sessions are set up correctly and that the monitoring interfaces on the AppResponse appliance are receiving data.
b)
Note the Packet Loss (Inbound and Outbound) values in the top table.
If these values are high, you might be seeing one-way traffic only in your spans. Check the span configuration, and the duplex settings of both the monitoring interfaces and the switch.
c)
If you are using AppResponse Rover, make sure the Rover can see bi-directional traffic.
d)
Note the Severity indicators next to each of the metrics. These are based on adaptive thresholds.
Note—When you first set up an AppResponse appliance, throughput and traffic are always zero, so these alarms are always red for the first few days. After AppResponse has run for a week or so, these adaptive indicators become accurate.
3)
Click on the Top Applications tab in the bottom pane. This shows a pie chart of the top 10 applications seen by the appliance.
Figure 2 “Total Traffic Dashboard” Insight > “Top Applications” Tabbed Page (EXAMPLE)
 
By default, AppResponse categorizes applications by standard IANA port number and protocol (such as HTTP over port 80 in Figure 2). Any unrecognized applications are displayed by port number and protocol.
When you click on the Top Applications tab, what applications do you see?
4)
Click on the Top /24 Subnets tab to see the traffic throughput in each subnet.
Figure 3 “Total Traffic Dashboard” Insight > “Top /24 Subnets” Tabbed Page (EXAMPLE)
 
In Figure 3, the 10.3.40.0/24 subnet has the highest throughput. What subnets do you see when you click on this tab? Have you defined any Business Groups based on these subnets?

When you have some applications and business groups, the Application Dashboard Insight is a great place to get some high-level performance metrics for the top applications and user groups in your network.
Perform the following steps:
1)
From the AppResponse Insights window, open the Application Dashboard from the Insight Index.
The Application Dashboard Insight enables you to view key metrics for a specific application. This section describes some of these metrics.
Figure 4 “Application Dashboard” Insight (EXAMPLE)
 
2)
In the Top 5 Applications bar chart (top-left pane), select one of the bars.
–> The other panes are updated with relevant information for the selected application.
3)
In the bottom pane, click through the Internal IPs, External IPs, IP Conversations, and Business Groups tabs.
These pages show the top talkers for the application by throughput. These pages are useful for identifying the internal and external resources that use the selected application most heavily.
4)
In the bottom pane, click the Response Time Composition tab.
Figure 5 Response Time Composition Chart in “Application Dashboard” Insight (EXAMPLE)
 
The Response Time Composition Chart (RTCC) is one of the most useful diagnostic charts in AppResponse. You can immediately identify the individual factors that contribute most to the total application response time for the application.
The application in Figure 5 has three significant components of delay: connection setup time (blue), server response time (red), and payload transfer delay (green). What are the highest components of delay for your application? Is there one component that qualifies as a “primary bottleneck?”

Defining Alerts for Performance Issues
You can use AppResponse to establish baselines of key performance metrics for different user groups, and then define alerts on the values of those key performance metrics. Alerts are useful for
Notifying you and others of abnormal or problematic behavior.
Generating notifications proactively of existing or potential problems before they impact a significant number of users.
Determine whether a perceived problem (by end users) is the result of an actual problem in the network or in one or more applications.
In the following sections, you will use one of the Baseline Insights to determine a threshold for acceptable performance in your network. Then you will define an alert to notify you when this performance threshold is exceeded.
Performance Baselines and Alert Thresholds
The Baseline Insights are useful for trending key performance metrics over different time periods. You can use these Insights to answer questions such as:
How does a key performance metric—such as user response time, latency, server response time—change over a specific time period? How does this trend compare to a similar time period from the past?
Is the value of a metric trending upward or downward over a specific time period?
What are the baseline values for my key performance metrics, and where should I set my alert thresholds?
By studying the trends of key performance metrics, you can determine “normal” vs. “abnormal” behavior for key applications and Business Groups. After you establish baseline behaviors for key performance metrics, you can create alerts that notify you when specific applications or Business Groups are experiencing sub-optimal performance.
Do the following steps:
1)
In the Insight Index, click the Baseline tab.
2)
Click the Trending Yesterday / Last Week / Last Month link to open this Insight.
–> The Trending Yesterday / Last Week / Last Month Insight appears. This Insight shows the top ten Business Groups by Throughput (Inbound and Outbound) over the project time. The Throughput (Inbound and Outbound) metric is selected by default.
Figure 6 “Trending Yesterday / Last Week / Last Month” Insight (EXAMPLE)
 
3)
In the Top Business Groups table (left), select a group.
–> The Time Series graph (top right) shows the throughput for the selected group over the current project time.
–> The Compare Time Series graph (bottom right) shows the throughput for the equivalent time period yesterday (the Yesterday’s Interval tab is selected by default).
Note that, in both graphs, a solid horizontal line shows the mean throughput. You can use this line to see the overall trend for the selected metric over the time period of interest.
4)
In the Metric pull-down menu (top center), select a metric for which you would like to define a threshold for acceptable network or application performance. (In the next section, you will define an alert that gets triggered whenever this threshold is exceeded.)
Hint—If you are not sure which metric to select, we recommend the Application Performance > User Response Time (Servers) metric.
This metric measures the average response time for all servers in the selected Business Group, as experienced by the clients that send requests to those servers1. For this reason, User Response Time (Servers) is one of the most useful metrics for measuring response times as experienced by end users. By trending this metric over time, you can obtain a good baseline for defining thresholds for acceptable response times.
5)
In the Top Business Groups table, click through the groups until you find a group for which you would like to define an alert threshold.
6)
Determine an acceptable threshold for the Business Group and metric of interest. In general, you want to define a threshold that represents the highest (or lowest) threshold for acceptable performance, so that alerts are generated only when the performance is clearly “out of bounds.”
You will use this threshold to define an alert later in this tutorial, so you might want to write down the metric name and threshold value now.
Hint—The following section provides a real-world example of how you might determine an alert threshold.
Determining an Alert Threshold: Example
In the following figure, an AppResponse user has selected the “NC Office” Business Group and the User Response Time (Servers) metric. The Time Series graph (top right) shows that today, the overall server response time spikes up to 12,000 msec but averages around 4000 msec. The same time period a week ago suggests an average response time a little higher—around 5000 msec. Based on the these results, the AppResponse user decides to set an Alert on User Response Time (Servers) for the NC Office Group between 4000 – 6000 msec.

Defining the Alert
In this section, you will define an alert based on the threshold you determined in the previous section.
Perform the following steps:
1)
In the Desktop Console, choose Tools > Alert Manager…
–> The Alert Manager appears. You can use this window to define alerts for any Business Group and metric in the Desktop Console.
2)
Click New (bottom left) and enter a name and description for the alert you want to define.
Figure 8
Alert Manager (EXAMPLE)
 
3)
Specify the following options in the Identity tabbed page:
–
Name
–
Description
–
Type = Static
–
Dashboard Section—This setting determines where triggered alerts will appear in the Dashboard (View > Dashboard in the Desktop Console). You can specify Client, Server, or Network. In general, you want to set this option based on the type of metric tested by the alert. User Response Time (Servers) is a server-related metric, for example, while Packet Retransmission Rate and Packet Loss are network-related.
Figure 9
Alert Manager > “Identity” Tabbed Page (EXAMPLE)
 
4)
Click the Group tab and select the Business Group that you want the alert to monitor.
Figure 10
Alert Manager > “Group” Tabbed Page (EXAMPLE)
 
5)
Click the Condition tab and specify the following:
–
The metric you want the alert to test
–
A Minor, Major, and/or Critical threshold for the metric (you must specify at least one threshold)
Figure 11 Alert Manager > “Condition” Tabbed Page (EXAMPLE)
 
Note that you can define an alert that tests complex threshold conditions, combined with the boolean operators AND and OR.
6)
Click the Schedule tab and specify when and how often the alert is tested. This page has the following fields:
–
Days (Days when the alert is run)
–
Time (The time window when the alert is run)
–
Interval (How often the alert is run during the specified time window)
Figure 12
Alert Manager > “Schedule” Tabbed Page (EXAMPLE)
 
7)
Click the Actions tab and define the actions to run when the alert is triggered. You can specify one or more of the following actions:
–
Email (Send an email to one or more addresses)
–
SNMP (Send an SNMP trap)
–
Syslog (Generate a Syslog entry)
In Figure 13, the AppResponse user has configured the Desktop Console to send an email to two addresses when the alert is triggered.
Figure 13
Alert Manager > “Actions” Tabbed Page (EXAMPLE)
 
Note the “on crossing” and “continuous” radio buttons. This also determines how often an action is generated:
–
If “on crossing” is selected, the action is performed only when the threshold is first met or exceeded.
–
If “continuous” is selected, the action is performed whenever the alert is run (as specified in the Schedule tabbed page) and the threshold has been met or exceeded. (This setting will usually result in more frequent actions.)
8)
Click the Links tab. This page allows you to define the Insight links that are available when viewing episodes for this alert. When a user examines an episode in the Dashboard Manager, these links provide a quick way to launch relevant Insights with the Business Group monitored by that alert.
Figure 14
Alert Manager > “Links” Tabbed Page (EXAMPLE)
 
9)
To apply your changes and save the alert, click Apply.

Enabling colleagues to view reports from a central web server, without the need to access the Desktop Console.
Emailing reports directly and automatically to selected colleagues

Creating an SLA Dashboard
The SLA Dashboard is a good way to visualize the end-user experience for multiple applications, business groups, or locations. This window is useful for answering questions such as:
Which applications are having problems in real time?
Which groups of users are being affected by poor performance?
Is a problem localized to a specific site or group, or a specific application?
Does the problem occur continuously, or only at certain times (during morning logins, for example, or when users return from lunch?)
How many users are being impacted by poor performance?
In the following figure, the dashboard shows a set of Applications (rows) and Business Groups for users in different geographic locations (columns). The light in each cell indicates the SLA status for that application and Business Group: green (in compliance), yellow (approaching threshold), and red (exceeded threshold).
Figure 22 SLA Dashboard (EXAMPLE)

Take-Away Points
This section summarizes the high-level points discussed in this tutorial.
Viewing Overall Traffic Levels
You can use the Audit Insights to identify the applications on your network and the heaviest users of those applications.
The Total Traffic Dashboard displays high level metrics about total network utilization and performance.
You can use the Total Traffic Dashboard to
–
Identify top applications by throughput for your entire network.
–
Identify top subnets by throughput for your entire network
–
Identify applications that are consuming network resources, and the clients and servers involved.
Very low traffic and high levels of packet loss might indicate problems with the span configuration, port configuration on the appliance, or port problems on the switch.
Viewing High-Level Performance Metrics
You can use the Application Dashboard Insight to
–
See key performance metrics for your largest applications.
–
Identify the top sources and destinations for specific applications.
You can use the Response Time Composition Chart to identify the primary causes of delay for specific applications.
Performance Baselines and Alert Thresholds
The Baseline Insights are useful for trending metrics over time and comparing metrics from different time periods.
The Trending Yesterday / Last Week / Last Month Insight is useful for quickly trending any metric over different time periods for the same business group.
Use the Baseline Insights to trend key performance metrics and establish a baseline value for that metric. After you establish a baseline, you can create one or more alerts to track any deviation from that baseline.
Defining the Alert
You can use alerts to notify you of abnormal behavior proactively.
An alert can be static (hardcoded) or adaptive (dynamic).
You can set up alerts to run continuously or only during specific times.
You can define alerts for many different kinds of key performance metrics.
It is good practice to trend and baseline your metrics before you establish alert thresholds.
Creating and Publishing Reports
You can instantly publish a snapshot of almost any Insight or report you are looking at in the Desktop Console.
Snapshot reports are a great way to share information quickly.
You can generate reports manually or on an automated schedule. You can either email reports or post them to the Reports page in the web UI.
Posting and emailing reports is a good way to share data without giving other users access into the Desktop Console.
Creating an SLA Dashboard
You can use SLA Dashboards to identify key application performance issues across several Business Groups.
You can define different thresholds for an SLA to identify the severity of an issue (in compliance, approaching threshold, or exceeded threshold).
The SLA Dashboard is a great operations tool to visualize the performance of specific applications across multiple Business Groups in your operational network.
1
In this context, a “client” is any host that sends a TCP request message and a “server” is any host that responds to a TCP request message.
